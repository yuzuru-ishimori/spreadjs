## 1. この文書の目的

本書は、Excelライクな業務入力インターフェースを自社開発する構想について、これまでの壁打ちで出た問題意識、判断、採用方針、対象機能、対象外機能、技術的な懸念、未決定事項を記録するための文書である。

ChatGPT Pro等へ開発計画の作成を依頼する際の入力資料としても使用するが、単なるプロンプトではなく、今後の設計判断の根拠を残すことを主目的とする。

---

## 2. 背景と問題意識

Excelは、業務データの入力インターフェースとして非常に完成度が高い。

一般的なWebフォームと異なり、Excelでは次の操作を高速に行える。

- セル単位・範囲単位の選択
- キーボード中心の連続入力
- 複数行・複数列のコピー＆ペースト
- 行・列の追加、削除、移動
- 複数セルへの一括入力
- フィルハンドルによる連続入力
- Undo／Redo
- 並べ替え、フィルター
- 書式や数式を含む表形式編集

Webアプリでも同様の入力体験を求められることは多いが、単にHTMLの表や入力欄を並べただけでは、Excelと同等の操作感にはならない。

特に日本語利用者にとっては、セル選択直後のIME入力、変換確定Enterとセル確定Enterの区別、セル移動後のIME継続などが不自然だと、実用上大きなストレスになる。

このため、既存のWebアプリごとに簡易的なExcel風画面を個別実装するのではなく、長期的に複数案件で再利用できる自社製のスプレッドシート型入力基盤を構築する。

## 3. 製品コンセプト

### 3.1 目指すもの

TypeScriptで実装された、業務入力向けのリアルタイム共同編集スプレッドシートコンポーネントを自社開発する。

主な特徴は次のとおり。

- Excelに近いセル操作・キーボード操作
- 日本語IMEを最優先にした入力体験
- 数万行を扱えるCanvasベースの描画
- 複数人によるリアルタイム共同編集
- 他ユーザーの現在位置・選択範囲の可視化
- 基本的な数式
- 文字列、数値、日付、プルダウンを扱う業務入力
- ReactおよびHonoを利用するWebアプリへ組み込み可能
- グリッド本体はReactやHonoに依存しない
- 将来的に複数の自社Webアプリで再利用可能

### 3.2 目指さないもの

Excelそのものの完全な代替や完全互換製品は目指さない。

特に、Excel固有の高度な機能やファイル互換性よりも、業務Webアプリに必要な入力操作、共同編集、データ統制、API連携を優先する。

---

## 4. 外部ライブラリに対する方針

今後長期間利用し、複数案件で共通基盤として使うことを想定しているため、SpreadJS、Handsontable、Univer、AG Grid等のスプレッドシート・データグリッド製品への中核依存は避ける。

理由は次のとおり。

- ライセンス条件や価格変更の影響を受けたくない
- 製品の廃止、仕様変更、提供元の方針変更に左右されたくない
- 日本語IMEや業務固有操作を自社判断で修正したい
- 複数の自社アプリへ自由に組み込みたい
- 中核部分の技術知識と制御権を自社に保持したい

ただし、一般的な低レベルライブラリや開発補助ツールまで完全に排除するものではない。

依存を避けたい対象は、製品の中核動作を支配するスプレッドシートライブラリである。

## 5. 技術構成の基本方針

### 5.1 採用方針

- 言語：TypeScript
- フロントエンドアプリケーション：React
- API・Webサーバー：Hono
- ビルド：Viteを有力候補とする
- グリッド描画：Canvas 2D
- セル編集：常駐textareaをCanvas上へ重ねる方式
- 通信：HTTP API＋WebSocket
- 重い処理：必要に応じてWeb Worker
- グリッドコア：React、Hono、特定UIフレームワークから独立

### 5.2 Reactの担当範囲

Reactは次のようなアプリケーション外周を担当する。

- 画面レイアウト
- ツールバー
- ダイアログ
- 設定画面
- ファイル・表の選択画面
- 利用者・権限表示
- 業務アプリ固有UI

セルデータ、選択状態、描画、IME入力、Undo／Redo等のグリッド中核状態をReactのstateで直接管理しない。

### 5.3 グリッドの公開形態

グリッド本体は命令型APIを持つ独立ライブラリとして設計する。

```ts
const grid = new NanairoGrid(container, options);

grid.setDocument(document);
grid.on('change', handleChange);
grid.on('selection-change', handleSelectionChange);
grid.destroy();
```

その上に、必要に応じて次を提供する。

- Web Componentラッパー
- Reactラッパー
- Hono連携サンプル

## 11. 日本語IMEに関する最重要要件

### 11.1 基本方針

日本語IMEの操作感は、可能な限りExcelと同じ感覚を目指す。

利用者に細かな仕様を指定させるのではなく、Excel利用者が自然に期待する操作を基準にする。

### 11.2 期待する操作

- セルを1回クリックした直後から日本語入力できる
- 矢印キーで移動した直後から日本語入力できる
- Enter移動後も日本語入力できる
- Tab移動後も日本語入力できる
- 最初の1文字が半角英字として取り残されない
- IME変換確定のEnterではセル移動しない
- 変換確定後に再度Enterを押した場合はセル移動する
- 変換途中でエディターDOMが作り直されない
- 変換途中で値が外部から再設定されない
- 変換途中でCanvas再描画が発生しても入力を失わない
- 変換途中でスクロールしても入力欄が正しいセルへ追従する
- Microsoft IMEとGoogle日本語入力で実用的に動作する
- macOSの日本語入力についても検証対象とする

### 11.3 実装方針

セルごとにinput要素を生成する方式ではなく、グリッド全体で1個のtextareaを常駐させる。

- セル選択時にtextareaを破棄・再生成しない
- textareaの位置とサイズだけを変更する
- 選択中もネイティブ入力要素としてフォーカス可能な状態を維持する
- compositionstart、compositionupdate、compositionend、beforeinput、inputを扱う
- KeyboardEvent.isComposing等を利用する
- IME変換中はセル移動、確定、値整形、再マウントを行わない

### 11.4 IMEと共同編集の分離

IME変換中の未確定文字列は、他ユーザーへセル値として配信しない。

共同編集へ送る情報は次のように分ける。

- IME変換中：セルを編集中であるというPresenceのみ送信
- 変換中文字列：ローカルのみで保持
- セル編集確定：確定値を操作として送信

同じセルへリモート更新が到着しても、IME変換中のtextareaを即座に上書きしない。

## 13. リアルタイム共同編集

### 13.1 基本要件

リアルタイム共同編集は初期設計から含める。

後付けではなく、データモデル、操作モデル、Undo／Redo、数式参照、行列操作を共同編集前提で設計する。

### 13.2 オフライン方針

長時間のオフライン編集はサポートしない。

短時間の通信断からの再接続や未送信操作の再送は検討するが、長時間オフラインで大量編集し、後から自動マージする機能は対象外とする。

そのため、初期版ではCRDTを必須とせず、オンライン接続を前提としたサーバー主導型同期を有力方針とする。

### 13.3 サーバー主導型同期

想定する流れは次のとおり。

1. クライアントが操作を生成する
2. Hono WebSocketサーバーへ送信する
3. サーバーが権限・リビジョン・対象存在を検証する
4. サーバーが操作へ確定リビジョンを付与する
5. 操作ログへ保存する
6. 接続中クライアントへ配信する
7. 定期的にスナップショットを生成する

操作例：

- SetCellValue
- SetCellFormula
- SetCellStyle
- InsertRows
- DeleteRows
- MoveRows
- InsertColumns
- DeleteColumns
- MoveColumns
- SetColumnWidth
- SetRowHeight
- MergeCells
- UnmergeCells
- SortRange

### 13.4 行列の固定ID

行番号、列番号だけをデータの識別子にしない。

```ts
type RowId = string;
type ColumnId = string;
```

画面上の行番号・列番号は、行順序・列順序から算出する表示座標とする。

これにより、共同編集中の行挿入、削除、移動、並べ替えでも、セル、数式、選択位置を安定して参照できるようにする。

## 14. 他ユーザーのカーソル・選択位置表示

### 14.1 要件

複数人が同じ表を開いた場合、誰がどのセルまたは範囲を操作しているか分かるようにする。

初期版で表示したい情報は次のとおり。

- 他ユーザーのアクティブセル
- 他ユーザーの選択範囲
- 他ユーザーが編集中のセル
- ユーザーごとの識別色
- 氏名、表示名、または短いラベル
- 接続中ユーザー一覧

### 14.2 Presenceとして分離

カーソル・選択位置は、セル値を変更する永続操作とは別のPresence情報として扱う。

Presenceの特徴：

- WebSocketで一時配信する
- 操作ログへ永続保存しない
- 再接続後に現在状態を再送する
- 切断時に一定時間後に消す
- 送信頻度を制限する
- 選択移動のたびにDBへ保存しない

例：

```ts
interface UserPresence {
  userId: string;
  displayName: string;
  colorToken: string;
  activeSheetId: string;
  activeCell?: {
    rowId: string;
    columnId: string;
  };
  selection?: {
    startRowId: string;
    startColumnId: string;
    endRowId: string;
    endColumnId: string;
  };
  editingCell?: {
    rowId: string;
    columnId: string;
  };
  updatedAt: number;
}
```

### 14.3 初期版のカーソル定義

初期版では、テキストエディター内の文字単位キャレット位置まで共有しない。

共有対象は次に限定する。

- アクティブセル
- 選択範囲
- 編集中セル

IME変換中文字列や変換候補は共有しない。

## 15. 競合処理

### 15.1 同一セルの同時編集

初期案：

- 他ユーザーが編集中のセルを可視化する
- 原則としてソフトロックとし、編集自体は禁止しない
- 確定操作はサーバー受付順または確定ルールに従う
- 上書きされた操作もログへ残す
- 利用者へ競合発生を通知する

ハードロック方式との比較はPoCで行う。

### 15.2 行削除とセル編集

例：

- 行削除がサーバーで先に確定した場合、その行へのセル更新を拒否する
- セル編集が先に確定した場合、その後に行削除を適用する
- 拒否された入力値を失わせず、利用者が確認・コピーできる形で保持する

### 15.3 行列操作と選択位置

行・列の追加、削除、移動が発生した場合、固定IDに基づいて各ユーザーの選択位置を維持する。

対象行または列が削除された場合の選択移動規則は別途定義する。

## 16. Undo／Redo

共同編集では、ドキュメント全体を過去状態へ巻き戻さない。

Undoは、自分が行った操作を打ち消す新しい操作として扱う。

ただし、他ユーザーによる後続操作がある場合、単純な逆操作が妥当でないことがある。

開発計画では少なくとも次を設計対象とする。

- セル値変更のUndo
- 貼り付けのUndo
- 行列追加・削除のUndo
- 書式変更のUndo
- 他ユーザー更新後のUndo
- 削除済み対象に対するUndo
- 競合時の利用者通知

初期リリースでは、対応できる操作を限定することも許容する。

---

## 17. 永続化

初期方針は、操作ログとスナップショットの併用とする。

- 操作ログ：監査、再生、共同編集同期、障害解析
- スナップショット：高速な初期読込
- Presence：永続化しない

検討事項：

- スナップショット生成間隔
- 操作ログ保持期間
- 文書バージョン
- トランザクション境界
- 大量貼り付けを1操作とするか複数操作とするか
- 数式計算結果を保存するか再計算するか
- サーバー側の正規値
- 切断時の未送信操作

## 22. 成功条件

本製品が実用に耐えるかどうかは、機能数よりも次の操作品質で判断する。

- 日本語入力がExcel利用者にとって自然である
- 1回クリック後にすぐ日本語入力できる
- 連続入力時にフォーカスやIMEが途切れない
- 数万行でもスクロールと選択が滑らかである
- コピー＆ペーストが業務入力で安定する
- 他ユーザーの位置が分かり、共同編集に不安がない
- 同時編集や通信断で入力内容が黙って消えない
- Undo／Redoが予測可能である
- Hono＋Reactの複数アプリへ再利用できる
- 特定の外部スプレッドシート製品に依存しない

## 絶対条件

- TypeScriptを使用する
- Reactを業務アプリ外周に使用する
- HonoをAPI・WebSocketサーバーに使用する
- グリッドコアはReactおよびHono非依存とする
- 描画はCanvas 2Dを基本とする
- セル編集は常駐textarea方式を基本とする
- 数万行をサポートする
- 日本語IMEを最優先要件とする
- Excelに近いセル移動・編集操作を目指す
- リアルタイム共同編集を初期設計から含める
- 他ユーザーのアクティブセル、選択範囲、編集中セルをPresenceとして表示する
- オンライン接続を前提としたサーバー主導型同期を基本とする
- 長時間のオフライン編集は対象外とする
- 行・列は固定IDで管理する
- 操作ログとスナップショットで永続化する
- 数式は簡易機能から段階的に拡張する
- 文字列、数値、日付、プルダウンを初期業務入力型とする
- 外部スプレッドシートライブラリへ中核依存しない
