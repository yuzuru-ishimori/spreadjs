## 目的

`@nanairo-sheet/grid` Facade を **collaboration: false（単独グリッドモード）** で成立させる。共同編集サーバー（Node）なしで mount でき、確定値の保存を**利用側アプリの API に接続する公開契約（イベント/Command）**経由で行えることを実証する（憲章 §11.1 の実証・S2-6・DD-026 統合①の前提）。

## 背景・課題

- Stage 1（DD-016）で Facade は**共同編集経路のみ実証済み**。現状 `mount()` は `serverUrl` 必須で、boot=/config 取得→WS 接続→`SessionSync`（ClientSession）に描画・IME・イベントの全配線が依存する（`packages/grid/src/mount-controller.ts`）。collaboration off の経路は存在しない＝**本DDの支配的リスク**。
- 最初の実 consumer = housing-e-kintai-next（React 19 + Vite 8 + TS / FastAPI(Python) + PostgreSQL・売上入力画面）。バックエンドが Python のため共同編集サーバーを立てず**単独グリッドモード先行**（ユーザー決定 2026-07-16。調査: `doc/archived/DD/DD-023/consumer-candidates.md`）。
- DD-023 Phase 1 DAレビュー#2: 「保存経路が collaboration 前提の内部配線に依存していれば統合初日に露見」— この懸念の潰し込みが目的。
- Beta 製品境界（roadmap §6）: 単独グリッドモードでは**認証・保存の責務は全面的に利用側アプリ**。SDK は Command/Event 契約のみ提供する。

## 検討内容

詳細比較は Phase 1 成果物 `doc/DD/DD-024/standalone-contract.md` へ（50行超は添付）。論点と仮説:

1. **Options の形（collaboration off の指定方法）**: 案a=判別 union（`GridMountOptions = 共同編集用 { serverUrl, ... } | 単独用 { mode: 'standalone', columnOrder 必須, ... }`。serverUrl×単独の排他を型で担保・fail-fast 実装が最小）／案b=憲章 §11.1 字面どおり `capabilities: { collaboration: false }` フラグ＋実行時検証。仮説=案a（憲章のコード例は意図の例示であり字面拘束しない）。→ **要確認①**
2. **確定値の保存イベント契約（cell-commit 相当）**: IME確定・Delete 等で committed になった変更を `GridEvent` の新種別（例 `{ type: 'cell-commit', changes: [{ rowId, columnId, value, previousValue }] }`・1確定操作=1イベント=batch 単位）で通知。**保存失敗時の扱い**: 初版は「通知のみ・grid は書き戻さない（利用側が再注入で戻す）」か「grid 側 revert Command を持つ」か。仮説=通知のみ（責務境界＝roadmap §6 と整合・最小契約）。→ **要確認②**
3. **初期データ注入（setDocument 相当）**: mount options の `initialData`（静的注入）のみか、mount 後の再注入 Command（`setDocument`/`setCells`）まで含めるか。housing は react-query 5 でサーバー状態を非同期取得するため mount 後注入が実運用ではほぼ必須と推定。仮説=両方をスコープに含める（mount 後注入がないと DD-026 で手戻り）。→ **要確認③**
4. **内部方式**: 案A=ClientSession をローカル完結 transport（loopback/no-op）で再利用（楽観適用・DocumentView 配線を共有・分岐最小）／案B=mount-controller から collab 依存を剥がし単独用 document ホルダーを新設。Phase 1 で設計判断（Codex レビュー対象）。
5. **共同編集専用面の off 時挙動**: `connection`/`pending`/`rejected`/`divergence` イベントは発火しない・`connectionState()` の返り値定義・`displayName`/`clientId` 等 server 系 options の混在は config error で fail-fast——を契約として明文化する。
6. **DX 成果物**: `doc/quick-start.md` へ単独モード節・保存接続の参照実装サンプル・`apps/showcase/src/features.json` 更新（AGENTS.md の更新義務）。

## 決定事項

**要確認①〜③はユーザー回答済み（2026-07-16・dd-auto 仕様確認ゲート）**:

- **① Options の形 = 案a（判別 union）**: serverUrl との排他を型で担保。憲章 §11.1 のコード例は意図の例示と解釈。
- **② 保存失敗時 = 通知のみ**: grid は書き戻さない（利用側が再注入で戻す）。roadmap §6 の責務境界と整合。
- **③ 初期データ注入 = mount 時＋mount 後再注入の両方**: housing の react-query 非同期取得を想定し setDocument 相当を含める。

Phase 1 の Human Spec Gate は**方向確定済みのため差分確認へ短縮**（standalone-contract.md は①〜③を前提に型定義・イベント schema・fail-fast 条件を文書化し、E2E シナリオ合意と内部方式 A/B の設計判断を残す）。

- **スコープ外**: React Facade（DD-025）／実 consumer への統合そのもの（DD-026）／Clipboard・行操作・列タイプ（DD-020/021/027）／共同編集⇔単独の実行時切替（mount 時固定）／単独モードでの数式。
- 単独グリッドモードにおける認証・保存の責務は全面的に利用側アプリ（roadmap §6 を契約文書へ転記する）。
