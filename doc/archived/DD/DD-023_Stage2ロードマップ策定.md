## 目的

Stage 1 SDK Alpha 完了（DD-018・2026-07-15）を受け、Stage 2（社内SDK Beta）の DD ロードマップ `doc/plan/phase2-dd-roadmap.md` を策定して正式版へ昇格する。あわせて Stage 3（実案件採用）の移行条件から逆算した「Stage 3 準備条件」を同ロードマップに組み込み、Stage 2 の終わり（exit）がブレない構造を作る。

## 背景・課題

- `stage2-backlog.md` は意図どおり**バックログ（送り項目の記録）**であり、ロードマップ化は「Stage 2 開始時」のタスクとして未実施のまま残っている（同文書ヘッダ・DD-018 要確認D）。
- バックログは Stage 1 からの送り項目のみで、**憲章側が Stage 2 に要求する項目が未統合**: §15 Stage 2 移行条件（2アプリ利用・API差分監視・移行ガイド・回帰の継続実行・Testkit/診断・Plugin/Adapter 実案件検証）、§26.3（deprecation policy・bundle size budget・docs site・利用状況計測）、§16.1（KPI 正式化）。
- **期限到来の未決定事項**: P-01 製品名・P-14 リポジトリ名（「Stage 1 リリース前」＝超過）、P-06 versioning・P-10 非推奨期間・P-12 Telemetry（「Stage 2 前」）、P-07 Plugin API v1（「2案件目開始前」）。
- **状況変化＝実 consumer 候補の具体化**: `C:\repo\housing-e-kintai-next`（不動産業務 勤怠・営業管理システムの Streamlit→React 19+Vite+FastAPI リエンジニアリング。調査記録=`DD-023/consumer-candidates.md`）への組み込み意向をユーザーが表明（2026-07-16）。React consumer のため roadmap §7 昇格条件により **React Facade が必須化**。バックログ §3.6 の列タイプ体系（ReadyCrew 案件DB 相談）と合わせ、Stage 1 計画時に存在しなかった要件系が優先順位を変えうる。

## 検討内容

- 先例踏襲: DD-007（Phase 3 で phase1-dd-roadmap 採択・外部レビューで昇格）・DD-018（stage2-backlog 確定）の構造を Stage 2 に再適用する。
- Stage 3 計画の形: フル計画書は実案件（3件以上）が未確定な現時点では推測になるため作らない。**S2 移行条件のハードゲート表＋「Stage 3 準備条件」節＋薄い `stage3-outlook.md`（1〜2ページ）**の2層とする（ユーザー合意 2026-07-16）。
- 共同編集サーバーの同居問題: housing-e-kintai-next のバックエンドは FastAPI（Python）。共同編集を使う場合 Node 製 collaboration-server の別プロセス同居が必要。単独グリッドモード（憲章 §11.1・collaboration: false）の成立性は Stage 1 で未実証のため、consumer 統合方式の確定が Phase 1 の要確認となる。

## 決定事項

Phase 1 要確認への回答（ユーザー・2026-07-16）:

1. **最初の consumer 対象画面** → 回答: housing-e-kintai-next の**売上入力画面**（移行計画書 PoC 候補〔歩合計算/勤怠入力〕とは別の画面。要件調査を統合DDの先頭タスクとする）
2. **2つ目のアプリ** → 回答: **ReadyCrew 案件DB**（商談進捗パイプライン画面・列タイプ体系の出所）
