## 目的

Stage 1 SDK Alpha の縦切りDD群（DD-012〜018）が乗る**共通基盤を機械的に設置**する（roadmap §3 DD-011）:
package skeleton／package boundary lint（`apps/*` 間・内部相対 import の恒久禁止）／contract test 骨格／常設不変条件スイート runner（§2.3）／independent consumer harness（§7）／Phase 1 用 DD 差分テンプレ改修（Risk Class ヘッダ＋製品化6観点）。

- **依存: 子DD DD-011-1（packageリネーム）→ 本DD**（要確認④のユーザー決定 2026-07-13＝「今 rename する・子DDへ切出し」）。本DDの skeleton・boundary lint・contract test は **rename 後の最終名 `@nanairo-sheet/{types,core,collab,server,formula}`（ディレクトリも `packages/{types,core,collab,server,formula}` へ統一済み）の上に構築**する。前提訂正: scope 付与は D-003 で実施済みのため rename の実体は `sheet-` プレフィックス除去＋ディレクトリ名統一（詳細は `DD-011-1_packageリネーム.md`）。

## 検討内容

1. **boundary lint の実装方式**: ESLint flat config の `no-restricted-imports`/カスタム判定（R1/R2/R3/R5）＋専用スクリプト（R4 の package root 脱出判定・R7 の AST/型情報検査。境界文書 §4.2 注記どおり **R7 は正規表現のみの実装は不可**）。R6 は既存 `typecheck:core`（ADR-0022）を流用し新規実装しない。
2. **不変条件スイートの置き場**: (a) 専用 workspace package を新設 ／ (b) ルート `tests/` ディレクトリ＋既存 vitest から実行。→ **(b) を採用**。workspace 追加を最小化し package-lock 変更を抑える（並行セッション lock 安全）。既存 `packages/*/src/test-support.ts`（collab/server）と `sim/remote-update-simulator.ts`（台帳 Q6: DD-011/013 で再判断）を素材に流用。
3. **consumer harness の形態**: 独立プロジェクト（**workspaces 対象外**のディレクトリ。`npm pack` tarball からインストール）。monorepo workspace link・source path 直接参照は S1-3 不合格条件のため使わない。実在社内アプリへの統合は DD-016 の担当。→**要確認③＝雛形で確定**（2026-07-13 ユーザー回答）。
4. **DD差分テンプレ改修**: `doc/templates/` は `dd-update.sh`（dd-know-how からの Pull 更新）の上書き対象のため、直接改修は DD-008 要確認(b) で一度**非採用**とした先例がある。製品化6観点は既に `doc/plan/phase0-dd-roadmap.md` に存在。反映先の選択肢: (a) `doc/templates/` を直接改修（次回 dd-update で差分衝突リスク）／(b) プロジェクト管理のヘッダ雛形を `doc/plan/` 等 dd-update 非管理の場所へ新設し各DDがコピー／(c) 上流 dd-know-how へ反映（本リポジトリ外作業）。→**要確認①＝(b) で確定**（2026-07-13 ユーザー回答。templates 直接改修はしない・DD-008 先例と整合）。
5. **package rename（`sheet-*` プレフィックス除去 → DD-009 論理名）**: DD-009 は「判断と実行は DD-011」とした（強制はしていない）。今 rename すると全 import・package-lock の大変更（並行セッションと衝突しうる）。見送ると lint は論理名↔現行名マップで運用（境界文書 §4.3 移行期方式）＝後日 rename 時に lint 設定・baseline・contract test の二重手戻り。→**要確認④＝「今 rename する」で確定**（2026-07-13 ユーザー回答）。**子DD `DD-011-1_packageリネーム.md` へ切り出し済**（`@nanairo-sheet/sheet-*` → `@nanairo-sheet/{types,core,collab,server,formula}`＋ディレクトリ名も統一）。これにより**論理名↔現行名マップは不要**になり、本DDは最終名の上に直接構築する。新設 Facade は最初から `@nanairo-sheet/grid`・`@nanairo-sheet/server-hono`（D-003 整合）。
