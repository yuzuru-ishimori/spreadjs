# 背景と論点：開発プロセスの「密度」をこれからどうするか

> ChatGPT向けの自己完結ブリーフィング（2026-07-12）。この1本で文脈が閉じるように書いてある。詳細の裏付けは `03`（DDロードマップ）・`04`（DD-005実物）・`05`（計画書抜粋）を参照。

---

## 1. 何を作っているか / なぜ難しいか

- **プロダクト**：業務Webアプリに組み込む、TypeScript製の**リアルタイム共同編集スプレッドシート型入力基盤**。React（業務UI外周）／独自グリッドコア（Canvas 2D・依存ゼロ）／Hono＋WebSocket（API・同期）／PostgreSQL。
- **技術的に難しい3点**（ここが成立するか不明なので先にPoCしている）：
  1. **日本語IME**：Canvas上のセル編集で、先頭文字欠落・変換確定順序（isComposing中確定=順序A／compositionend後=順序B）などブラウザー差が出る。プロジェクトの最大リスク（R-01）。
  2. **数万行Canvas描画**：50,000行×200列を仮想スクロール・固定行列・高DPIで実用速度で描く。
  3. **サーバー主導の共同編集**：CRDTではなくサーバー全順序シーケンサー。楽観適用＋rollback/replay、セル単位の楽観的競合検知（beforeRevision）、入力を黙って上書きしない。
- これら3つを**一つのセル編集フローとして結線**するのが直近の DD-005（統合PoC）。

## 3. 開発方式（DDプロセス）

- **DD（設計書）** 単位で開発。**1 DD = 1レビューサイクル**。巨大な親DDを作らず、受け入れ基準を独立検証できる単位で分割する。
- **文書3層**：ロードマップ（DD間の依存・順序・Go/No-Go）／個別DD（1レビューサイクルの実装・検証）／ADR・仕様書（長期の設計判断）。
- **1本のDDの流れ**：起票 → **仕様確認ゲート（1回）** → 実装 → レビュー → 検証 → 完了。「いきなりコードを書かない」。
- **AIエージェントへの自動振り分け（dd-auto）**：起票＝Fable → 〔仕様確認〕→ 実装＝Opus → Codexレビュー。
- **自動継続ルール**（人間ゲートを減らす既存の工夫）：合意済みスコープ内なら、ファイル分割・型詳細化・テスト追加・リファクタ・DA/Codex指摘対応は**止まらず自動継続**。
- **停止して人間判断を仰ぐのは限定条件のみ**：受け入れ基準の変更／対象範囲の増減／ユーザー操作の変更／データ消失につながる競合仕様の変更／ADR基本方針の転換／外部依存の新規導入／Go・No-Go判断／明示的な実機ゲート。
- **Phase 1以降は縦切り**：技術領域ごとではなく「利用者が一連の操作を完了できる単位」でDDを切る（各DDがCanvas・IME・共同編集・数式・Undoを縦に貫く）。加えて製品化6観点（公開API／Facade境界／再利用性／Adapter・Plugin拡張性／DX成果物／API互換）を各DDで確認する。

## 4. 現在の「密度」の実像

1本のDDに、以下が積み重なっている：

- **レビュー4層**：
  - 😈 **DA（悪魔の代弁者）**：自己批判レビュー。各Phaseで実施し、発見・重要度・再現手順・対応を表に記録。
  - 🧑‍⚖️ **Codex自動コードレビュー**：`effort xhigh`（最高）。重いDDでは2回（抽出差分＋統合差分）。
  - 🗣 **外部レビュー（ChatGPT）**：手動で実施し、指摘・採否・反映先をDDへ還流。
  - 🖐 **実機ゲート**：ユーザーが実ブラウザー・実IMEで手動検証（syntheticは実IMEの代替にならないと計画書が明記）。
- **厚い証跡**：実装ログ（Phaseごと）・受け入れ基準対応表・エビデンス文書・計測レポート・スクリーンショット多数・実IMEトレース。
- **細かいPhase分割**：1本のDDの中を Phase 0〜5 に割り、各Phaseに「📐実装前詳細化 → 実装 → 🔬機械検証（test/typecheck/lint/build/doc-check/E2E）→ 😈DA」を課す。
- **モデル使い分け**：起票=Fable（軽量・安価）／実装=Opus（高性能）／レビュー=Codex xhigh。

## 5. 具体例：DD-005（統合PoC）がなぜ重くなったか

- **役割**：Phase 0で唯一「IME・Canvas・共同編集を1本に結線」するDD。3領域が同じフロー上で初めて衝突する「最も衝突しやすいDD」。
- **やったこと（貼り合わせに留まらない）**：新パッケージ `sheet-collaboration` の抽出、**状態所有権モデルの確定**（ClientSessionだけを文書の唯一の正本にし、Canvas・IMEに第二・第三の文書状態を作らない）、セル単位beforeRevision競合検知、RowId安定化。**本来Phase 1でやる製品アーキ設計を前倒しで固めている**。
- **密度の実数**：内部Phase 0〜5、DA 18項目、Codex xhigh 2回（2回目で5指摘）、外部レビュー7指摘、外部アドバイス12点採用、**テスト438＋E2E17**、証跡・スクショ多数。Phase 1〜4実装完了、残りは実機ゲート（Phase 5）。
- **密度のROI（＝無駄ではない証拠）**：
  - headed実機スモークが**ユニット緑なのに実行時破綻するバグを2件検出**（CORS未設定で接続不能／初回ロードで空Axisに例外→50,000行が描画されない）。
  - Codex xhighが**5件指摘**（reject後のCanvas無効化漏れ、presence専用メッセージで再描画されない、等）。うち4件修正、1件を既知制約化。
  - 「syntheticだけで実IME成立を偽装しない」ための歯止めをDAが監視。
- **つまり**：密度は実際に問題を捕まえている。論点は「その密度のROIが、**より低リスクなPhase 1以降でも見合うか**」。

## 6. 実行体制の現実（ここが助言の前提）

> 【要確認：相談者が実態に合わせて補正すること】

- **人間はほぼ1名**（プロジェクトオーナー兼ゲート担当）。計画書の「3〜4名体制」ではなく、**単独開発＋AIエージェント**で回している。
- **実装・レビューの多くをAIが担う**：起票=Fable、実装=Opus、コードレビュー=Codex、外部レビュー=ChatGPT。夜間に自律でDDを進める運用もある。
- **希少資源＝人間の確認時間とAPI/サブスク枠**。人月見積（AIなし・複数名前提）とは律速が違う。人間がボトルネックになるのは「仕様確認ゲート」「実機ゲート」「Go/No-Go判断」「レビュー結果の取捨」。
- 「人を増やして速くする」は選べない。**減らせるのは各DDのゲート回数・確認負荷・レビュー層・証跡量・Phase分割の細かさ**。

## 7. 密度を左右する7つのレバー（現状値つき・中立）

| # | レバー | 現状 | 薄くする方向の例 |
|---|--------|------|----------------|
| L1 | **DDの粒度** | 1 DD = 1レビューサイクル。細かく分割 | 縦切りDDを大きめに束ねる／レビューサイクルを機能群単位に |
| L2 | **レビュー層の数** | DA＋Codex＋外部＋実機の最大4層 | 低リスクDDは層を減らす（例：DA＋Codexのみ、外部は節目だけ） |
| L3 | **Codexのeffortと回数** | xhigh固定・重いDDは2回 | リスクに応じて effort を下げる／1回に集約／差分が小さい回は省略 |
| L4 | **PoC・実装の作り込み度** | DD-005は製品アーキを前倒しで作り込み | PoCは成立実証に絞り、アーキ確定はPhase 1へ送る |
| L5 | **証跡・ドキュメントの厚さ** | 実装ログ・DA表・エビデンス・スクショ・トレース | 節目のみ厚く、通常DDは要点＋機械検証ログに圧縮 |
| L6 | **ゲート／停止条件の頻度** | 仕様確認1回＋自動継続＋限定停止条件 | 実機ゲートをまとめ打ちに／仕様確認を非同期承認に |
| L7 | **モデル使い分け** | 起票=Fable/実装=Opus/レビュー=Codex xhigh | 低リスク作業は軽量モデル、重要局面だけ高性能モデル |

## 8. Phase 0 と Phase 1以降で最適点が違う可能性

- **Phase 0（検証）**：間違った結論（「成立する」と誤判定）が最も高くつく。ここでの高密度は**保険として妥当**な面が強い。実際にバグ・設計欠陥を捕まえている。
- **Phase 1以降（構築）**：土台の成立性は確定済み。作るのは「検証済みの原理」の製品化。同じ密度が同じROIを生むかは不明。**過剰なら速度を大きく損なう**。
- 相談の核心は「Phase 0で確立した重い型を、そのままPhase 1以降へ持ち込むべきか、フェーズに応じて調整すべきか」。

## 9. 密度を落とすべきでない領域（品質クリティカル）

薄くする議論でも、ここは慎重にすべき候補（相談者の現時点の考え。ChatGPTの評価も歓迎）：

- **IMEの正しさ**：先頭文字欠落・composition・確定順序A/B。最大リスクR-01。**実機ゲートはsyntheticで代替不可**と計画書が明記。
- **共同編集の収束とデータ消失防止**：beforeRevision競合・rollback/replay・単一正本・入力を黙って上書きしない、という不変条件。
- **公開API/互換性**（Phase 1以降）：SDKとして外部提供する契約。破壊的変更は後で高くつく。

## 10. コスト構造（何を減らすと何が浮くか）

- **人間の確認時間**：ゲート回数・レビュー結果の取捨・実機試験。← 最も希少。
- **AIの実行コスト（API/サブスク枠）**：Codex xhigh・Opus実装・多数のテスト実行。effort・回数・モデル選択で増減。
- **壁時計時間（納期感）**：Phase分割の細かさ・レビュー層・実機ゲート待ちで伸びる。
- **やり直しコスト（密度を下げすぎた場合）**：見逃したバグ・設計欠陥が後段（Phase 2/3）で顕在化すると、修正は指数的に高くつく。← 密度のROIの源泉。
