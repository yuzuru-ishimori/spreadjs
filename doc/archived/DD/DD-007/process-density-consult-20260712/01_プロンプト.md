# ChatGPTへ貼り付けるプロンプト

---

あなたには、**ソフトウェア開発のプロセス設計・デリバリー戦略のアドバイザー**として相談に乗ってほしい。特に「開発者ほぼ1名＋AIエージェント（設計書起票・実装・コードレビューをAIが担当）」という現代的な体制での、**プロセスの重さ（密度）と速度・品質のトレードオフ**に明るい前提で答えてほしい。

## 状況（詳細は添付を参照）

- リアルタイム共同編集スプレッドシート基盤（TypeScript・Canvas描画・日本語IME・サーバー主導共同編集）を、**Phase 0（技術成立性のPoC）**で開発中。
- 開発は **DD（設計書）** 単位で回している。**1 DD = 1レビューサイクル**が原則。各DDは「起票 → 仕様確認ゲート → 実装 → レビュー」を通る。
- レビューが**多層**になっている：DDごとに **DA（悪魔の代弁者による自己批判）＋ Codexによる自動コードレビュー（最高effort、時に2回）＋ 外部レビュー（ChatGPT）＋ 実機ゲート（ユーザー手動）**。加えて厚い証跡（設計ログ・エビデンス・スクショ・トレース）。
- 直近の **DD-005（統合PoC）が想定より重くなった**：内部を Phase 0〜5 に細分し、DA 18項目、Codexレビュー xhigh 2回（2回目で5件指摘）、外部レビュー7指摘、テスト438＋E2E17、証跡多数。まだ実機ゲート（Phase 5）が残っている。
- 疑問：**Phase 0が重かったからといって製品の土台がほぼ出来たわけではなく、実装の山場（Phase 2共同編集・Phase 3業務MVP）はこれから**。この密度のままPhase 1以降も進めると、各Phase 8〜10週の見積りをさらに押し上げるのではないか。

## 相談したいこと

**これからの進め方（Phase 0の残り〜Phase 1以降）を、今の密度のまま行くべきか、もう少し薄くできるか。** 薄くできるなら、どのレバーを・どのPhaseで・どこまで調整すべきか。今のままが良いなら、その根拠は何か。

## 出してほしい成果物（この順で）

1. **現状診断**：今の密度は「過剰／適正／不足」のどれに近いか。**Phase 0（検証フェーズ）とPhase 1以降（製品構築フェーズ）を分けて**評価してほしい。Phase 0で密度が高いこと自体は妥当か、それとも Phase 0 の中でも過剰だったか。
2. **密度レバーごとの具体提案**：添付『02_背景と論点』の「密度を左右する7つのレバー」それぞれについて、**現状維持／軽量化／強化**のどれを、**どのPhaseで**推奨するか。軽量化する場合は「何をやめる・減らす・条件付きにする」を具体的に。
3. **推奨の総括**：結局「今のまま」寄りか「軽量化」寄りか、**一番効くレバー2〜3個**を優先度つきで。判断の理由も。
4. **軽量化する場合のガードレール**：密度を下げても品質・データ整合・IME正しさを守るために、外してはいけない歯止め（添付の「密度を落とすべきでない領域」を踏まえて）。
5. **見落としている観点**：この体制・このプロダクトで、私（相談者）が見落としていそうなリスクや、逆に過剰に心配している点。

## 制約・お願い

- **抽象論（「アジャイルにしましょう」等）ではなく、添付に書かれた実際のプロセスに即して**具体的に。トレードオフは必ず明示してほしい（「速くなるが○○を失う」）。
- **体制の前提を踏まえること**：実装・レビューの多くをAIエージェントが担い、**希少資源は人間（ほぼ1名）の確認時間とAPI/サブスク枠**。「人を増やす」は選べない。人間のゲート回数・確認負荷を減らす方向の提案を重視してほしい。
- **フェーズで最適解が変わる前提**：Phase 0（未知を潰す＝間違うと高くつく）とPhase 1以降（検証済みの土台の上に作る）で、適切な密度は違うはず。その違いを明示的に扱ってほしい。
- 断定できないところは仮定を置いた上で述べてよい。ただし仮定は明示すること。
