



# 結論

**推奨は「一律に今のまま」ではなく、明確に軽量化寄りです。**  
ただし、IME・共同編集・永続化・公開APIなどの高リスク領域は、現在に近い密度を維持します。

評価を分けると次のとおりです。

| 対象 | 評価 |
|---|---|
| Phase 0全体 | **適正〜やや過剰** |
| DD-005の技術検証密度 | **適正** |
| DD-005の文書・レビュー運用密度 | **やや過剰** |
| Phase 1で現在の密度を標準適用 | **過剰** |
| Phase 2の共同編集・永続化 | **再び高密度が必要** |
| Phase 3以降 | 機能リスクに応じた選別が必要 |
| Phase 6 | リリース品質として高密度に戻す |

密度はPhaseの進行に従って単純に下げるのではなく、次のような波形にするべきです。

> **Phase 0：高 → Phase 1：中 → Phase 2：高 → Phase 3：中〜高 → Phase 4・5：中、一部高 → Phase 6：高**

# 1. 現状診断

## 1.1 Phase 0の密度

Phase 0を高密度にした判断自体は妥当です。

このプロジェクトでは、次の誤判定が極めて高くつきます。

- 実IMEでは成立しないのにsynthetic testだけで成立したと判断する
- 各クライアントが一時的に動いたことを「収束した」と誤認する
- Canvas単体の性能を、共同編集・IME統合後も維持できると仮定する
- PoCの簡易状態管理を製品アーキテクチャへ持ち込む
- 競合時に利用者入力を失う経路を見逃す

DD-005では、実ブラウザー試験によって、全ユニットテストが成功している状態でもCORSと初期Axis構築順序の2件が検出されました。またCodexレビューでは、reject後の再描画、Presence単独更新、行挿入後の位置再解決など、実際の不具合につながる指摘が得られています。したがって、DD-005のレビューが単なる儀式だったとは評価できません。fileciteturn0file1 fileciteturn0file3

特にDD-005は、IME・Canvas・共同編集という3つの独立PoCを初めて単一フローへ結線するDDです。このDDだけは、通常のPhase 0 DDより重くなる合理性があります。

### ただし、すべてが必要だったわけではない

有効だったのは主に次の部分です。

- 実ブラウザーでのheaded smoke
- 実WS・複数クライアントを通したE2E
- 状態所有権の明文化
- cell-level beforeRevisionの検証
- Codexによる独立したコードレビュー
- 実IMEをsyntheticで代替しないゲート

一方、次は圧縮可能です。

- DD内部の各Phaseで毎回DAを実施する
- 同じ事実をヘッダ、進捗、実装ログ、DA表、エビデンス文書へ重複記載する
- 全Phaseに詳細な実装前設計と完了ログを残す
- 外部レビュー、DA、Codexをほぼ同じ範囲へ重ねる
- スクリーンショットを多数保存し、文章でも同じ結果を再記述する

実際、DD-005では冒頭ステータスに「434 tests」とあり、後段では「438 tests」となっています。これは品質上の重大問題ではありませんが、同じ進捗数値を複数箇所へ転記する運用が既に同期不良を起こしている例です。**証跡を厚くすると正確になるとは限らず、正本が増えて古くなるリスクも増えます。** fileciteturn0file3

## 1.2 「1 DD = 1レビューサイクル」とDD-005

DD-005は、実質的には少なくとも次の2つのレビュー単位を含んでいます。

1. `sheet-collaboration` 抽出というパッケージ境界変更
2. IME・Canvas・共同編集の統合フロー

Codexレビューを2回必要としたことも、この2つが異なるレビュー対象だったことを示しています。

DD-005を今から分割し直す必要はありません。しかし今後は、

> **異なる差分範囲についてCodexレビューを2回必要とするなら、そのDDは2つに分ける候補**

という分割基準を追加するべきです。

## 1.3 Phase 1以降の密度

現在の密度をPhase 1以降の全DDへそのまま適用するのは過剰です。

ただし、「Phase 1以降は検証済みだから低リスク」という理解も正確ではありません。

Phase 1では、PoCコードを製品コードへ変換する過程で、次の新しいリスクが発生します。

- 公開APIの固定
- Facadeと内部パッケージの境界
- エラー処理・破棄・再初期化
- 複数案件での再利用性
- 長期保守できる型と拡張ポイント
- PoCコードを採用するか、再実装するか
- パフォーマンス最適化と抽象化の衝突

Phase 2はさらに高リスクです。sequencer、operation log、snapshot、rollback/replay、再接続、idempotency、永続化が入るため、DD-005以上に後戻りが高くつく領域があります。フェーズ計画でもPhase 2が共同編集Alphaと永続化の中心とされています。fileciteturn0file4

したがって、正しい方針は「Phase 1以降を薄くする」ではなく、

> **通常DDは薄くし、不可逆・データ損失・外部契約に関係するDDだけ高密度へ戻す**

です。

## 1.4 Phase境界に一つ重要な不整合がある

DDロードマップでは、Phase 1最初のDDを、

> 日本語で入力し、サーバー保存され、他クライアントへ反映され、Undo登録される

縦切りとして定義しています。fileciteturn0file2

一方、開発計画書ではPhase 1は「単一利用者コア」であり、共同編集・サーバー保存・Presence・rollback/replayはPhase 2です。fileciteturn0file4

これは軽微な表現差ではありません。

- Phase 1を本当に単一利用者製品化とするのか
- DD-005の共同編集基盤を利用し、Phase 1から最小共同編集を含めるのか
- UndoをPhase 1でどこまで実装するのか

をDD-007で確定しないと、Phase 1の期間、DD粒度、レビュー密度を決められません。

# 2. 7つの密度レバーごとの提案

## L1：DDの粒度

### 推奨

- Phase 0残り：**現状維持。ただしDD内部の細分化は軽量化**
- Phase 1：**通常DDは少し大きく束ねる**
- Phase 2：**高リスク単位で細かくする**
- Phase 3以降：**機能群によって使い分ける**

Phase 1以降のDDは、次を基本単位とします。

> **一つの利用者成果＋一つの支配的リスク**

「セルを連続入力できる」は妥当ですが、そこへ共同編集、Undo、公開API、React wrapper、DX成果物をすべて入れると再びDD-005化します。

分割シグナルは次のいずれかです。

- 異なる差分にCodexレビューが2回必要
- 人間の設計判断ゲートが2回以上必要
- 主要な状態所有者が複数変わる
- 一つのDDで公開APIと永続化方式を同時に確定する
- 受け入れ基準の一部を独立してリリース・検証できる

逆に、CSS、メニュー、表示ラベル、サンプル更新などは複数を束ねてよいです。

**トレードオフ**：DDを大きくすると起票・レビューの固定費は減りますが、差分の理解と原因切り分けは難しくなります。

---

## L2：レビュー層の数

### 推奨

- DD-005：**現在のままPhase 5を完了**
- DD-006：**DA＋Codexを基本。外部レビューは結果が曖昧な場合のみ**
- DD-007：**DA＋外部レビュー＋人間Go/No-Go。コード変更がなければCodexコードレビュー不要**
- Phase 1通常DD：**DA＋Codex**
- Phase 2重要DD：**DA＋Codex＋対象別の実機・障害試験**
- 外部レビュー：**Phase境界、ADR転換、公開API確定、Go/No-Goに限定**

現在の4層を全DDへ適用する必要はありません。

特に外部レビューは、コードレビューよりも次の用途に絞ると価値が上がります。

- 仕様の抜け
- アーキテクチャの責務境界
- フェーズ計画
- 公開API
- 技術負債の受容判断
- Go/No-Go判定

一方、実機ゲートはレビュー層ではなく、**自動化不能な受け入れ試験**です。IMEやブラウザー実行時挙動では省略できません。

**トレードオフ**：レビュー層を減らすと異なる観点からの指摘は減ります。ただし、同じ仕様と同じ差分を複数モデルが読むだけでは、独立性は限定的です。

## L3：Codexのeffortと回数

### 推奨

| リスク | Codex |
|---|---|
| IME、sequencer、protocol、rollback/replay、永続化、公開API、schema migration | **xhigh 1回** |
| 通常の機能DD | **highまたはmedium 1回** |
| UI、ドキュメント、サンプル、挙動保存リファクタ | **medium、または複数DDをまとめてレビュー** |
| 2回目 | **条件付き** |

2回目のCodexレビューは次の場合に限定します。

- 1回目の指摘修正で設計または差分範囲が大きく変わった
- P1相当が複数見つかった
- 修正が別モジュールへ波及した
- レビュー対象が「抽出」と「統合」のように本質的に異なる
- 未解決findingが残る

単に「重いDDだから2回」ではなく、**差分の性質が変わったか**を条件にするべきです。

**トレードオフ**：effortを下げれば深い状態遷移バグの検出力は落ちます。ただし、低リスク差分へxhighを使う費用対効果は低いです。

## L4：PoC・実装の作り込み度

### 推奨

- DD-005：作り込みは結果的に妥当
- DD-006：**成立性・計測・方式決定に限定して軽量化**
- Phase 1：**現在のPhase exitに必要な製品品質まで**
- 将来Phaseの機能・拡張性：**先回りしない**

DD-006では、次を作れば十分です。

- CellStore方式比較
- 代表データでのメモリ計測
- 最小formula parser
- 固定ID参照
- 依存グラフ
- replay時の決定性と性能
- 採用・不採用理由

この段階で、完全なPlugin API、Formula Worker、全関数体系、洗練された公開APIまで作る必要はありません。

Phase 1以降も「将来必要だから」という理由でPhase 3〜5の抽象化を先に実装すると、PoCで避けたはずの過剰設計が製品フェーズで再発します。

**トレードオフ**：先回りを減らすと後で一部を作り直す可能性があります。その代わり、未確定要件へ抽象化コストを払わずに済みます。

## L5：証跡・ドキュメントの厚さ

### 推奨：大幅に軽量化

通常DDの完了成果物は、次の5点で十分です。

1. スコープ・対象外・リスク区分
2. 受け入れ基準対応表
3. 機械検証結果の要約と生ログへの参照
4. findingと対応・既知制約
5. ADR・公開API・互換性への影響

削減対象は次です。

- Phaseごとの長文実装ログ
- 同じテスト件数の複数箇所への転記
- 全DA項目の文章化
- 自動テストで証明済みの画面を何枚も保存する
- 「変更なし」の6つの製品化観点を毎回長文で記載する

製品化6観点は、次のような高速チェックでよいです。

```text
公開API影響: なし
Facade境界: 変更なし
再利用性: 問題なし
Adapter/Plugin: 対象外
DX成果物: 更新不要
互換性: Internalのみ
```

スクリーンショットは次の場合だけ残します。

- 実IME
- Canvasの見た目
- 競合表示
- Presence
- ブラウザー実行時のみ再現する問題
- 第三者が視覚的に判定する必要がある受け入れ条件

**トレードオフ**：詳細な作業過程の追跡性は落ちます。ただし、結論・証拠・既知制約の追跡性は維持できます。

---

## L6：ゲート／停止条件の頻度

### 推奨：人間ゲートを例外ベースへ変更

最も効果が大きいレバーです。

Phase 1正式バックログがDD-007で承認された後は、DDを次の3区分にします。

| 区分 | 人間の仕様確認 |
|---|---|
| A：高リスク | 必須 |
| B：通常 | 承認済みバックログ範囲なら自動開始 |
| C：機械的・低リスク | 自動開始、複数DDをまとめて確認 |

A区分にする条件は次です。

- IME状態機械、textarea、focus、selectionを変更する
- sequencer、protocol、rollback/replay、OCCを変更する
- 永続化、snapshot、migrationを変更する
- データ消失やサイレント上書きの可能性がある
- Stableな公開APIを変更する
- 外部依存を追加する
- 受け入れ基準・操作仕様を変更する
- 自動試験で判定不能な受け入れ条件がある

実機ゲートも、DDごとではなく変更トリガー方式にします。

例：

- IME関連コード変更あり：実機ゲート必須
- IME関連コード変更なし：Phase境界または数DDまとめて回帰確認
- Canvas描画ロジック変更あり：headed確認
- ドキュメントやサーバー内部変更のみ：IME実機ゲート不要

現在定義されている「停止すべき条件」は妥当なので維持します。変更すべきなのは、**すべてのDDに人間仕様確認を置く部分**です。fileciteturn0file2

**トレードオフ**：人間が各DDを事前に読む機会は減ります。代わりに、承認済みバックログの範囲外へ出ない機械的な停止条件が必要です。

## L7：モデル使い分け

### 推奨：役割固定からリスク・作業種別ルーティングへ

現在の、

- Fable＝起票
- Opus＝実装
- Codex xhigh＝レビュー

という固定配分は分かりやすい一方、低リスクDDにも高コストモデルを使いやすい構造です。

次の配分が妥当です。

| 作業 | 推奨 |
|---|---|
| テンプレート起票、既存DDからの展開 | 軽量モデル |
| 既知パターンのUI、テスト追加、単純Adapter | 軽量〜中量モデル |
| 状態遷移、並行処理、IME、永続化 | 高性能モデル |
| 機械的リファクタ | 軽量モデル＋既存テスト |
| 高リスクコードレビュー | Codex xhigh |
| 通常コードレビュー | Codex medium/high |
| Phase計画、Go/No-Go、API設計 | 高推論モデルまたは外部レビュー |

異なるモデルを使うこと自体を品質証拠にしてはいけません。品質を担保するのは、独立した不変条件、実行結果、障害注入、実機確認です。

**トレードオフ**：軽量モデルの利用範囲を広げると、局所的な実装品質や説明品質が下がる場合があります。失敗しても自動検出・巻き戻し可能な作業へ限定する必要があります。

# 3. 推奨の総括

## 最終判断

**Phase 0の残りは少し軽量化し、Phase 1からはリスク分類型へ移行するべきです。**

全面的な軽量化ではありません。

### 優先度1：L6 ゲートを例外ベースにする

人間がほぼ1名である以上、最大のボトルネックはコード生成速度ではなく、仕様確認待ちとレビュー結果の取捨です。

承認済みPhaseバックログ内のB・C区分DDは、人間の事前承認なしに進める運用へ変更するのが最も効きます。

### 優先度2：L2 外部レビューと実機ゲートを選別する

- 外部レビュー：Phase境界・Go/No-Go・API・ADR
- 実機：自動化不能な変更があった場合
- 通常DD：DA＋Codex

とします。

### 優先度3：L5 証跡を単一正本化する

DD-005のような重複記録を通常運用にしないことです。

テスト数、コミット、受け入れ結果、既知制約は一箇所だけを正とし、他文書は参照にします。

### L3は費用削減として有効だが、優先順位は次

Codex xhigh回数を減らすとAPI枠は節約できますが、人間の待ち時間を最も減らすのはL6とL2です。

# 4. 軽量化する場合のガードレール

## 4.1 不変条件テストは削らない

次はDDとは独立した常設スイートにします。

### IME

- composition中にtextarea.valueを書き換えない
- selectionを破壊しない
- textarea instanceとDOM親を置き換えない
- 順序A/Bの双方
- remote update・rollback/replay中もdraft不変
- syntheticと実IMEを混同しない

### 共同編集

- サーバー全順序とクライアント最終hash一致
- rollback/replay後の収束
- beforeRevision不一致でサイレント上書きしない
- reject時に利用者入力を保持する
- idempotency
- reconnect/catch-up
- RowId・ColumnIdの安定
- snapshot＋logからの復旧

### 公開API

- Stable／Experimental／Internalの区分
- API contract test
- protocol/schema version
- 破壊的変更検出
- 移行ガイド要否判定

レビュー文書を薄くしても、これらのテストは減らしてはいけません。

## 4.2 リスク区分をDDヘッダの必須項目にする

各DDに次だけを追加します。

```text
Risk Class: A / B / C
Risk Triggers:
Human Spec Gate: required / skipped
Codex: xhigh / high / medium / none
Manual Gate:
External Review:
Evidence Level: full / standard / minimal
```

これにより、密度が担当モデルの気分で変わることを防げます。

## 4.3 既知制約は必ずPhaseバックログへ送る

「今回は対象外」と書くだけでは不足です。

各制約に、

- 影響
- 放置期限
- 解消予定Phase
- 解消しない場合の製品制約

を付けます。

特にDD-005の次は、Phase 1または2で明示的に回収が必要です。

- client→server欠落時の完全再整列
- 行挿入後のローカル選択・Enter移動先の再ベース
- snapshotベース初期化
- 実IME変換中に対象行が削除された場合の挙動

## 4.4 性能回帰予算を持つ

DDごとにフルベンチマークを行う必要はありません。

ただし次を超えた場合だけ、本格再計測を発動します。

- 初期ロード経路変更
- Document State表現変更
- Axis再構築条件変更
- Canvas描画キャッシュ変更
- operation replay方式変更
- Formula依存グラフ変更
- 大量paste・sort・filter・行移動の追加

通常DDでは軽量スモーク、対象変更時だけフル計測とします。

## 4.5 Phase 3のpilotはレビューで代替しない

Phase 3終了条件には4週間の社内pilotがあります。これはDA、Codex、E2Eでは代替できません。fileciteturn0file4

レビュー密度を下げても、実業務での次の確認は残す必要があります。

- 利用者が入力を失ったと感じないか
- 競合UIが理解されるか
- 操作遅延が業務を妨げないか
- Undo／Redoが期待どおりか
- 障害時の復旧が運用可能か

# 5. 見落としている観点

## 5.1 Phase 1のスコープ不整合が、密度問題より先に解決すべき問題

前述のとおり、ロードマップのPhase 1縦切りは共同編集・サーバー保存・Undoを含みますが、計画書のPhase 1は単一利用者です。

この不整合を残したまま軽量化すると、「本来Phase 2のリスクをPhase 1の軽量プロセスで扱う」可能性があります。

DD-007でPhase境界を確定することが最優先です。

## 5.2 PoCコードの「採用・再実装・破棄」判定が必要

DD-005はPoCでありながら、既に製品パッケージ境界や状態所有権をかなり固めています。

Phase 1開始前に、PoC成果物ごとに次を判定すべきです。

| 判定 | 意味 |
|---|---|
| Adopt | そのまま製品コードとして採用 |
| Harden | 基本設計を維持して製品品質へ強化 |
| Rewrite | 仮説は採用するが実装は作り直す |
| Discard | 検証用として破棄 |

これを行わないと、「PoCだから作り直す」と「かなり作ったので流用する」がDDごとに揺れます。

## 5.3 AIレビューの誤りは相関する

Fable、Opus、Codex、ChatGPTが別モデルでも、同じDD、同じ前提、同じ設計文書を読むため、誤った前提を共有する可能性があります。

独立性が高いのは、モデルの数よりも次の方法です。

- randomized operation test
- property-based test
- fault injection
- process restart
- snapshot restore
- 実ブラウザー
- 実IME
- 実データ量
- 利用者pilot

DD-005で最も価値が高かったのも、文章レビューより実行時検証でした。

## 5.4 テスト数を品質指標にしない

438 testsという数は進捗の参考にはなりますが、品質の主要指標ではありません。

見るべきなのは、

- どの不変条件を守っているか
- 同じ経路を重複テストしていないか
- テストが本番配線を通っているか
- false positiveがないか
- 障害経路を通しているか
- 後続DDで回帰を検出できたか

です。

## 5.5 公開APIをPhase 1で固定しすぎる危険

各DDで公開API・互換性を確認する方針は正しいです。ただし、Phase 1で全APIをStableにすると、Phase 2の共同編集とPhase 3の業務MVPで設計変更が困難になります。

推奨は次です。

- Phase 1前半：Internal中心
- Phase 1後半：Experimental公開
- Phase 2：protocolとserver adapterをExperimental
- Phase 3 pilot後：主要APIをStable

互換性を重視することと、未成熟なAPIを早く固定することは別です。

## 5.6 プロセス改善を感覚で判断しない

Phase 1の最初の5 DD程度で、次を記録してください。

- DDの壁時計時間
- 人間確認時間
- ゲート待ち時間
- Codex/API消費
- レビュー層別のP1・P2 finding数
- 後続DDで発見された見逃し
- 文書更新時間
- 手戻り率
- flaky test数

その結果、例えば外部レビューが重要findingをほとんど出していないなら、Phase境界だけへ縮小できます。逆にmedium Codexで見逃しが増えるなら、対象領域だけhighへ戻せます。

# DD-007へ還流すべき決定案

1. **Phase 1の単一利用者／最小共同編集の境界を確定する**
2. 各DDへA・B・Cのリスク区分を付ける
3. B・C DDは承認済みバックログ内なら人間ゲートなしで開始する
4. Codex xhighはA区分のみ、原則1回
5. 外部レビューはPhase境界・API・ADR・Go/No-Goに限定する
6. 実機ゲートは変更トリガー方式にする
7. DD証跡を「AC表・検証結果・finding・既知制約・ADR影響」の5点へ圧縮する
8. Phase 1の最初の5 DDで密度変更の効果を計測し、運用を再調整する
9. PoC成果物についてAdopt／Harden／Rewrite／Discardを確定する
10. Phase 2では共同編集・永続化DDを再びA区分として扱う

**最終評価としては、DD-005は失敗例ではありません。高密度によって重要な問題を実際に検出した成功例です。**

ただし、その成功を「すべてのDDをDD-005と同じ重さにする根拠」として使うべきではありません。DD-005から抽出すべき標準は、文書量やレビュー回数ではなく、**単一正本、不変条件、実行時検証、実IME、データ損失防止という品質ガードレール**です。
