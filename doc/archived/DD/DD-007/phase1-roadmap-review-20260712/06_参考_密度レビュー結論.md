# 参考: 直前の密度レビュー（第3回外部レビュー）の結論

> 出典: `doc/DD/DD-007/chatgpt-review-20260712-3.md`（開発プロセスの「密度」をどうするかの相談結果）。**本 Phase 1 草案は、この結論をプロセス標準として織り込んでいる。** レビュー者が「前回何を勧め、今回の草案がそれをどう反映したか」を確認できるよう要約を添える。

## 1. 結論（密度レビュー）

- 推奨は「一律に今のまま」ではなく、**明確に軽量化寄り**。ただし IME・共同編集・永続化・公開API などの高リスク領域は現在に近い密度を維持。
- 密度は Phase 進行で単調に下げるのではなく**波形**にする: **Phase 0:高 → Phase 1:中 → Phase 2:高 → Phase 3:中〜高 → Phase 4/5:中（一部高） → Phase 6:高**。
- 正しい方針は「Phase 1以降を薄くする」ではなく、**「通常DDは薄く・不可逆／データ損失／外部契約に関わるDDだけ高密度へ戻す」**。

## 2. 7つの密度レバーの推奨（要約）

| レバー | 推奨 |
|---|---|
| L1 DD粒度 | 「一つの利用者成果＋一つの支配的リスク」。分割シグナル: Codex2回必要／人間設計ゲート2回以上／状態所有者が複数変わる／1DDで公開APIと永続化を同時確定／ACの一部を独立検証可能 |
| L2 レビュー層 | 通常=DA＋Codex。外部レビューは **Phase境界・API確定・ADR転換・Go/No-Go に限定**。実機ゲートはレビュー層ではなく自動化不能な受け入れ試験 |
| L3 Codex effort/回数 | A区分=xhigh 1回／通常=high or medium 1回／UI・doc=medium orまとめて。**2回目は「差分の性質が変わったか」を条件**に |
| L4 作り込み度 | 現在のPhase exit に必要な製品品質まで。将来Phaseの抽象化を先回りしない |
| L5 証跡 | **5点に圧縮**（スコープ/AC対応表/機械検証要約＋生ログ参照/finding・既知制約/ADR・API・互換性影響）。**単一正本**。スクショは実IME・Canvas見た目・競合・Presence・実行時のみ再現する問題に限定 |
| L6 ゲート頻度 | **人間ゲートを例外ベースへ（最も効く）**。承認済みバックログ内の B/C は自動開始。実機は**変更トリガー方式** |
| L7 モデル使い分け | 役割固定からリスク・作業種別ルーティングへ。**別モデルであること自体を品質証拠にしない** |

## 3. DD-007 へ還流すべき決定案（密度レビューが提示した10項目）

1. **Phase 1 の単一利用者／最小共同編集の境界を確定する**
2. 各DDへ A・B・C のリスク区分を付ける
3. B・C DD は承認済みバックログ内なら人間ゲートなしで開始する
4. Codex xhigh は A区分のみ、原則1回
5. 外部レビューは Phase境界・API・ADR・Go/No-Go に限定する
6. 実機ゲートは変更トリガー方式にする
7. DD証跡を「AC表・検証結果・finding・既知制約・ADR影響」の5点へ圧縮する
8. Phase 1 の最初の5DD で密度変更の効果を計測し、運用を再調整する
9. PoC成果物について **Adopt／Harden／Rewrite／Discard** を確定する
10. Phase 2 では共同編集・永続化DDを再び A区分として扱う

## 4. 削ってはいけないガードレール（密度レビュー §4）

- **不変条件テストは削らない**（IME・共同編集・公開API を DD独立の常設スイート化）。
- **リスク区分をDDヘッダ必須項目に**（Risk Class / Triggers / Human Gate / Codex / Manual Gate / External Review / Evidence Level）。
- **既知制約は必ずバックログへ送る**（影響・放置期限・解消予定Phase・未解消時の製品制約）。特に DD-005 の: client→server欠落の完全再整列／行挿入後の再ベース／snapshotベース初期化／実IME中の対象行削除。
- **性能回帰予算**を持つ（該当経路を変えたDDだけフル再計測）。
- **Phase 3 の4週間pilotはレビューで代替しない**。

## 5. 見落とし指摘（密度レビュー §5・草案に効くもの）

- **Phase 1 のスコープ不整合（単一利用者 vs 共同編集）が密度問題より先に解決すべき**（→草案 §1.2 で DD-010/011 分割として対応）。
- **PoCコードの Adopt/Rewrite/Discard 判定が必要**（→草案 DD-009 で対応）。
- **AIレビューの誤りは相関する**（同じ設計文書を読むため）。独立性は「モデルの数」より randomized/property-based/fault injection/実ブラウザー/実IME/実データ/pilot で担保。
- **テスト数を品質指標にしない**。
- **公開APIを Phase 1 で固定しすぎない**（Phase 1前半=Internal／後半=Experimental／Phase 2=protocol・adapter Experimental／Phase 3 pilot後=主要Stable）（→草案 DD-009 の成熟度方針で対応）。
