## 観点（この順で確認・指摘）

1. **正典との整合**: 草案が引用する計画書 §・憲章 §・Stage 1 移行条件・DD番号・既存DDの成果が、正典の実記述と一致しているか。**誤引用・存在しない節番号・事実と異なる要約**を洗い出す。
2. **フェーズ境界整合の健全性**: §1.2 の「計画書 §19 Phase 1/Phase 2 境界を DD-010/DD-011 の分割で保存する」という整合ロジックに穴はないか。Phase 2 相当リスク（永続化・OCC・reconnect）が Phase 1 の軽量枠へ漏れていないか。
3. **DD分割・依存の整合**: §4 の候補DD表と §5 の順序・依存に矛盾がないか（前提が後続に来ている等）。Risk Class の付与（A/B/C）が §2.1 のトリガー定義と整合しているか。過積載（1DDに支配的リスク複数）や、逆に縦切りになっていないDDはないか。
4. **Stage 1 移行条件の網羅**: S1-1〜S1-5 が担当DDへ漏れなく割り付いているか（§0 の表と §4 表の突合）。SDK Alpha 完了条件（consumer fixture・内部import禁止・公開export・Quick Start・React非依存）が最初の縦切りに反映されているか。
5. **既知制約の回収**: §6 の DD-005 引き継ぎ制約が、解消予定DD・未解消時の製品制約まで書かれているか。抜けている既知制約（密度レビュー §4.3 が挙げたもの）はないか。
6. **密度レジームの内部整合**: §2 の Risk Class・ゲート・Codex effort・証跡5点・実機トリガーが、密度レビュー結論（`chatgpt-review-20260712-3.md`）と齟齬なく写像されているか。
7. **未確定事項の妥当性**: §7 の論点が、計画確定に本当に必要な未決事項を捉えているか。決めずに進むと危険な未決事項の**抜け**はないか。
8. **文書整合**: DOC-MAP への登録・リンク・見出し構造・表の列ずれなど機械的な不備。
