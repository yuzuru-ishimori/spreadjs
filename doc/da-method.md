# DD + DA方法論：設計文書と悪魔の代弁者分析

## この文書の目的

AI支援開発（Claude Code等）において、**DD（Design Document）** と **DA（Devil's Advocate）分析** の組み合わせが手戻りの削減とバグ早期発見にどう寄与するかを解説する。

---

## 1. 概要

### DDとは

Design Document（設計文書）。機能追加やリファクタリングの **計画・設計・実装・検証** を1つの文書で管理する。

```
DD-001: 主菜A/B選択制
├── Phase 0: 設計判断（実装前の検証）
├── Phase 1: データ構造変更
├── Phase 2: プロンプト生成
├── Phase 3: レスポンス処理
├── Phase 4: UI実装
├── Phase 5: 品質検証
└── Phase 6: テスト・統合
```

### DAとは

Devil's Advocate（悪魔の代弁者）分析。各Phaseの前後で **「何が壊れるか」** を意図的に探す批判的レビュー。

| 従来のレビュー | DA分析 |
|-------------|--------|
| 「動くか確認する」 | 「どこが壊れるか探す」 |
| 「問題なさそう」で終了 | 「暗黙の前提が崩れるケースは？」と深掘り |
| 「コードが動いたからOK」 | 「半年後の変更で壊れないか？」と問う |

## 2. なぜDD + DAの組み合わせが強力か

### 2.1 Phase 0 DA：実装前に設計を壊す

最も価値が高いのは **Phase 0（実装前）のDA分析** である。

**実例：DD-001（主菜A/B選択制）Phase 0**

初期設計をDAで分析した結果、Critical 5件が発見された：

| 発見 | 問題 | 汎用パターン | もしDAなしだったら |
|------|------|-------------|-------------------|
| C1 | JSON構造が既存パーサーと非互換 | 新仕様と既存コードの非互換 | Phase 3で作り直し |
| C2 | 在庫チェックがA/B人数分離に非対応 | 既存ロジックが新要件の前提を満たさない | Phase 5で設計破綻 |
| C3 | 栄養スコアの分離方針が未定義 | 評価・集計ロジックの設計漏れ | Phase 5で追加設計 |
| C4 | 米の配置がA/B別料理系で未定義 | 共有リソースの競合・配分が未定義 | Phase 2で矛盾発覚 |
| C5 | servingsがstateに無くDOM直読み | 状態管理の不備（UIとロジックの分離不足） | Phase 1で全層波及 |

**5件すべてが実装前に修正された。** 各Phaseでの手戻りは発生しなかった。

> **教訓**: Phase 0 DAに30分-1時間投資することで、数時間-数日の手戻りを防げる。

### 2.2 深掘りDA：繰り返すほど深層の問題が出る

**実例：DD-003（プロンプト生成一本化）Phase 0 — 7ラウンド**

同じコードベースに対して、角度を変えながらDAを繰り返した結果：

```
Round 1: C1  ■             方針の妥当性
Round 2: C5  ■■■■■         3システム全差分比較（ピーク）
Round 3: C3  ■■■           ランタイム・インポートチェーン
Round 4: C3  ■■■           内部契約・既存バグ
Round 5: C3  ■■■           循環import・DOM依存
Round 6: C3  ■■■           sections契約・純粋関数性
Round 7: C0                収束確認
```

**Round 2が最大値になるパターン** が特徴的。初回は浅い探索、2回目で深い比較分析に切り替えた時にピークを迎え、以降は漸減する。

7ラウンドの累計: Critical 18件、Warning 35件、Info 6件（重複除く実質約55件）。

> **教訓**: 1回のDAで満足せず、Criticalが出続ける限り角度を変えて繰り返す。

### 2.3 修正後DA：同根パターンを潰す

**実例：DD-005（既存バグ修正）Phase 1 DA**

BUG-4（`state.bannedGroups = []` による参照切れ）を `state.bannedGroups.length = 0` に修正。修正後のDAで**同じパターンが別の2箇所にも存在**することを発見：

- `loadBannedGroupsTestPreset()` — プリセット読込時
- `utils.js loadSettings()` — localStorage復元時

1箇所の修正で安心せず、DAで「同根パターン」を検索することで見落としを防いだ。

> **教訓**: バグ修正後は「同じパターンが他にないか」のDA必須。発見した同根パターンは `doc/engineering-patterns.md` に昇格させ、DDアーカイブ後も横断参照できるようにする。

### 2.4 副産物としてのバグ発見

DD-003のDA（プロンプト統合の設計検証）を行う過程で、**DD-003とは無関係な既存バグ5件**が発見された：

| バグ | 深刻度 | 内容 |
|------|--------|------|
| フィールド名不一致 | Critical | Few-shot計算例の栄養値が常に0 |
| SERVINGS定数ハードコード | Warning | 5人設定で栄養値10倍膨張 |
| ファイルパス誤り | Warning | food_master.json読み込み失敗 |
| 配列参照切れ | Warning | クリア後の再追加で変更が反映されない |
| 全角括弧未正規化 | Warning | A/B分類ミスで栄養計算不正 |

これらは別DD（DD-005）に切り出して先行修正した。

> **教訓**: DAのスコープを「対象DDの設計検証」に限定せず、触れたコードの品質全般を見る。

### 2.5 DAが少なくて済むケースもある

**実例：DD-002（UI統合・タブ化）— 5 PhaseでCritical 0件**

DD-002はUIリファクタリング（8セクションを3タブに集約）。各PhaseでDAを実施したが、Critical 0件で完了した。これは「DAが無駄だった」のではなく、以下の条件が揃っていたため：

- **ロジック変更がない**: DOM構造の整理のみで、計算・検証ロジックに触れていない
- **テストが既にある**: 327件の既存テストが変更の安全性を担保
- **Phase 0で設計が単純**: タブ化は選択肢が少なく、設計判断の余地が小さい

> **教訓**: 全てのDDに深掘りDAが必要なわけではない。ロジック変更が少ない変更はPhase内DAのみで十分。

### 2.6 機械検証による嘘防止

DD では各Phase末尾に**機械検証タスク**を組み込む:

```
- [x] `npx tsc --noEmit` — TypeScriptエラーなし
- [x] `npm run build` — 成功
- [x] `npm test` — 全テストパス
```

LLMは「確認しました」と嘘をつきうるが、**コマンド実行結果のエビデンスを要求する**ことで嘘のつきようがなくなる。

機械検証 = 人間が判断しなくてもコマンド出力で合否が分かる検証:
- テスト実行、型チェック、lint、grep、ビルド、整合性チェッカー

目視が必要なもの（UXレビュー等）とは明示的に区別する。

> **教訓**: 各Phaseに最低1つの機械検証タスクを入れる。「テスト確認」ではなく「`npm test` → 全テストパス」のように期待結果まで書く。

### 2.7 DD検索でナレッジが検索可能に

DD が蓄積されると「あの機能、以前やったような気がするんだけど、どれだっけ？」という逆引きが課題になる。DD本体にはファイルパス・設計判断・gotchaが全て記録されているため、**grep による全文検索**でDD横断のナレッジベースとして機能する。

これにより、以前は「DD完了時にMEMORY.mdに転記する」という手作業（コンテキスト回収）が必要だったが、DD自体が検索可能なので転記が不要になった。

> **教訓**: DD本体に知見を書き込むことが最重要。MEMORY.mdへの転記は不要。

### 2.8 DAの限界

DAは万能ではない。以下の限界を認識しておく必要がある：

| 限界 | 具体例 |
|------|--------|
| **偽陽性が出る** | DD-001 Phase 3で「`{}`はfalsyだからバグ」と報告 → 実際はtruthy（言語仕様の誤解） |
| **重複発見が増える** | DD-003の7ラウンドで4件の重複。ラウンドが増えると新規発見の効率が下がる |
| **実行時の問題は見つけにくい** | 静的解析が中心のため、タイミング依存のバグや負荷問題は発見しにくい |
| **DAの質はレビュアーに依存** | 対象ドメインの知識がないと、表面的な指摘に終わる |

> **対策**: DAで「0件」になっても「見つけられなかった」可能性を常に意識する。テスト・E2E・実ユーザーフィードバックと組み合わせることで信頼性を補完する。

### 2.9 「件数」より「読み返す価値」

§2.1〜2.4 は「DAで多くのCriticalを発見した」実績を扱った。一方で、**指摘を量で評価すると質が下がる** という対称的な落とし穴がある。§2.5 が「0件が妥当なケース」を扱うのに対し、本節は「件数が多すぎてノイズ化するケース」を扱う。

**よくある推移パターン**

DAを実施すると、初回はしばしば 10〜20 件超の指摘が並ぶ。冷静に分類し直すと多くの場合、以下の混在が見られる:

- **意味あり**: 型・前提・スコープ判断・運用ルールに関わる項目
- **わかりきってる**: 実装に着手すれば自然に気付くレベルの規約・配慮事項

両者を冷静に分けると、後者が **半数以上を占める** ことが多い。これは「DAは批判レビューだから網羅的に出すべき」という暗黙のバイアスが働くためである。

**問題: Critical判定の閾値が相対的に下がる**

価値の低い指摘を多数Criticalに混ぜると、レビュー全体でのCriticalの意味が薄れる。後から読み返した時に「どれが本当に重要だったのか」が判別できなくなり、運用ルールとしての効力も失う。

**対策**

- §3.4 第4項目の「読み返す価値」基準で振い落とす
- 1件追加するごとに「これは次Phaseで自然に気付くか？」を自問する
- 「気付く」ならDAから外し、Phase実装時のチェックリストに格下げする

> **教訓**: §2.1〜2.4 の「Criticalを見逃さない」とセットで、「Critical判定の閾値を保つ」こともDAの質に直結する。件数を稼ぐDAは、ノイズが増えた瞬間に読み返す価値を失う。

## 3. 実践ガイド

### 3.1 DDの作り方

```markdown
# DD-{番号}: {タイトル}

## 目的（1-2文で）
## 背景・課題（なぜ必要か）
## 検討内容（選択肢の比較）
## 決定事項
## タスク一覧
  Phase 0: 事前精査 + DA
  Phase N: 実装 + DA
## ログ（作業記録）
## DA批判レビュー記録
```

**ポイント:**
- Phase 0を必ず設ける（設計判断とDA）
- 各Phaseの最後にDA批判レビューを入れる
- タスクが100行を超えそうなら分割を検討
- **DD完了前に実装中の知見（gotcha・回避策）をDD本体に書き戻す**

### 3.2 DA分析の実施方法

#### Phase 0 DA（設計段階 — 最重要）

| 観点 | 問うべき質問 |
|------|-------------|
| 矛盾・不整合 | DDの記述と実際のコードは一致しているか？ |
| 暗黙の前提 | 「当然こうなっているはず」と思い込んでいる箇所はないか？ |
| エッジケース | 入力が0、null、空文字、最大値の時に壊れないか？ |
| 依存関係 | 変更がどのファイルに波及するか、全て把握しているか？ |
| 将来の脆弱性 | 半年後に別の人がコードを変えた時、壊れやすい設計になっていないか？ |

#### Phase内DA（実装後 — 品質確認）

> **Codex併用時の省略**: 同PhaseでCodexレビューを実施する場合、Phase内DAはCodex findings（結果表と対応記録）で代替してよい。DAとCodexで同観点（認可・境界値・並行処理など）を二重に記録しない（DD-010-4方式）。

1. 変更ファイルを全行読み直す
2. 変更した関数名でgrepして呼び出し元を確認
3. 新規追加した関数のテストがあるか確認
4. **発見数の扱い**: 件数の下限（ノルマ）は設けない。§3.4 第4項目の基準を満たす指摘のみを記録し、それが0件なら0件が真の発見数である（§2.9参照）。ただし0件で完了とする前に、§2.5の「0件が妥当な条件」（ロジック変更が少ない・既存テストが充実・設計判断の余地が小さい）に照らして妥当性を確認すること。条件に当てはまらないのに0件の場合は、角度を変えてもう1巡してから判断する

#### 深刻度の判定基準

| レベル | 基準 | 例 |
|--------|------|-----|
| **Critical** | 今のPhaseでテストが落ちる or ユーザー操作で壊れる | 「パーサーが新構造を読めない」 |
| **Warning** | 特定条件で壊れうる（条件を具体的に書く） | 「5人設定の時だけ栄養値が膨張する」 |
| **Info** | 改善提案・将来のリスク | 「端数処理が不統一（実害なし）」 |

### 3.3 収束判定

| 条件 | アクション |
|------|-----------|
| Criticalが2ラウンド連続0件 | DA調査完了としてよい |
| 中レベル以上が1件以上 | 別角度で再チェック必須 |
| 重複発見が増えてきた | 収束の兆候。あと1ラウンドで打ち切り検討 |

### 3.4 DA品質フィルター（ノイズ削減）

DAレビューの精度を維持するため、以下のフィルターを適用する。

**1. 検証義務**: 指摘前にコード実行 or 仕様確認で事実を確認すること。「〜かもしれない」だけの推測指摘は記録しない。
- 「この変数は falsy になる可能性がある」（未検証） → NG
- 「L42 で `parseInt(value)` に空文字を渡すと NaN になる。実際に input を空にして確認済み」 → OK

**2. 既知事項の除外**: 以下は指摘対象外。
- コード内の `TODO` / `FIXME` コメントで明示されている事項
- DD の「Phase X で対応」と記載されている事項
- 別 DD にスコープアウト済みの事項

**3. 再現手順の必須化**: 深刻度「高」「中」には以下を必ず記載:
- **操作**: 何をすると（例: servingsB に 0 を入力して送信）
- **結果**: 何が起きるか（例: cuisineBPanel が非表示にならない）

**4. 「読み返す価値」基準**: 指摘は、後から読み返した時に判断の根拠として使える粒度のものに限る。具体的には以下のいずれかに該当するもの:

  1. **次Phase着手後に確実に詰まる地雷** — 仕様・スキーマ・型・前提条件を読み直して初めて気付くレベルの非互換・欠落
  2. **スコープ判断（やる / やらない）の合意が要る項目** — 本DDの境界を決める判断で、合意なしで進むと後で衝突する
  3. **過去の失敗の再発予防に効く運用ルール** — 同種の事故を繰り返さないための仕掛け

  以下はDA対象外（Phase実装時の自然な配慮で吸収できる）:
  - 実装中に当然行う規約レベルの配慮（命名・型変換・引数検証など、コードを書き始めれば気付く事項）
  - 「次Phaseで実測すれば足りる」レベルのメモ
  - 実害のない軽微な不整合

  品質指標は **件数ではなく「読み返した時に判断の根拠として使えるか」**。網羅性のために水増しすると、Critical判定の閾値が相対的に下がり、本当のCriticalが埋もれる（§2.9 参照）。

### 3.5 再チェック発動条件

以下のいずれかを満たす場合、別角度で再チェック必須:

| 条件 | 基準 |
|------|------|
| 中レベル以上の発見 | 1件以上 |
| 多数発見（小: 1-2ファイル） | 3件以上 |
| 多数発見（中: 3-5ファイル） | 5件以上 |
| 多数発見（大: 6ファイル以上） | 7件以上 |

**別角度チェックの方法:**
1. 視点を変える: 実装者 → レビュアー → ユーザー → 将来の保守者
2. チェック観点を変える: 機能正しさ → テストカバレッジ → 仕様書整合性
3. 関連ファイルを広げる: 直接変更 → import先 → 同ディレクトリの関連ファイル

### 3.6 Phase種別チェックリスト

**実装Phase:**

| チェック項目 | アクション |
|-------------|-----------|
| 関数網羅性 | 変更ファイルの全public関数をリスト、各関数のテスト有無を確認 |
| 呼び出し元 | 変更した関数名でgrep、呼び出し元への影響確認 |
| エッジケース | null, 空配列, 0, 負値, 境界値を渡した場合の動作確認 |
| 定数・マップ | 新規追加した定数/マップに漏れがないか全パターン突合 |

**テストPhase:**

| チェック項目 | アクション |
|-------------|-----------|
| 関数カバレッジ | export一覧とdescribe一覧を突合 |
| 新規関数 | 今回追加した関数すべてにテストがあるか |
| 異常系 | エラーケース・例外のテスト有無 |

**ドキュメントPhase:**

| チェック項目 | アクション |
|-------------|-----------|
| 仕様書同期 | 変更コードに対応する仕様書セクションの更新有無 |
| サンプル値 | ドキュメント内の例示値がコードと一致しているか |

## 5. アンチパターン

### やってはいけないこと

| アンチパターン | なぜダメか | 正しいやり方 |
|--------------|-----------|-------------|
| DA 0件を無検証で「問題なし」と報告 | 見落としている可能性が高い | §2.5の「0件が妥当な条件」と照合し、根拠を添えて報告する |
| 網羅性のために水増し指摘を出す | Critical判定の閾値が下がり、本当のCriticalが埋もれる | §3.4 第4項目で振い落とす |
| 「〜かもしれない」でCritical判定 | ノイズが増えてDAの信頼性が下がる | 事実確認してから深刻度を判定 |
| 既知の問題を再報告 | 既存TODO/DD記載事項の重複 | 既知事項はInfo扱いで参照先を明記 |
| 1回のDAで満足する | 深層の問題は角度を変えないと見えない | Criticalが出る限り繰り返す |
| Phase 0 DAを省略する | 最も費用対効果の高い工程を失う | Phase 0 DAは必須 |
| テスト漏れ | 新規関数を追加したがテストを書いていない | export一覧とテストdescribeを突合 |
| 呼び出し元未確認 | 関数の戻り値を変更したが呼び出し元を確認していない | 関数名でgrepして全呼び出し元を確認 |
| 定数マップ不完全 | 6種類あるはずが5種類しかマップにない | 全パターン列挙して突合 |
| 仕様書未更新 | コードは直したが仕様書に反映していない | 変更コードに対応する仕様書を特定 |

## 6. まとめ

DD + DAの組み合わせは、以下の3つの効果を同時に実現する：

1. **手戻り防止**: Phase 0 DAで設計段階の矛盾を発見し、実装前に修正する
2. **品質向上**: Phase内DAで実装の抜け漏れを検出し、同じPhase内で修正する
3. **既存バグ発見**: DAの副産物として、対象外のコードの品質問題も発見できる

**最も重要な原則**: Phase 0 DAに最もリソースを集中すること。実装後の修正コストは、設計段階の修正コストの数倍-数十倍になる。
