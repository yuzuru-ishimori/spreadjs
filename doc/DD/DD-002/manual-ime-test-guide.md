# DD-002 実機手動 IME 受入試験 手順書（Phase 6）

> 目的: 計画書 §18.1 の合格条件 1〜5 を、**Windows 11 実機の実 IME** で判定する。synthetic
> イベント（Phase 3 の状態機械ユニットテスト・Playwright）では実 IME の候補ウィンドウや

## 0. 対象環境（決定事項「試験対象環境」）

macOS・Firefox は本 PoC の対象外（Phase 0 後半以降へ送る）。Google 日本語入力が未導入の場合は

## 1. 起動と準備

1. リポジトリルートで `npm run dev`（Vite 開発サーバー）を起動し、表示された URL（既定
   `http://localhost:5173`）を対象ブラウザーで開く。
2. 画面下部の「IME イベントトレース」パネルの **IME 欄に、その環境の IME 名を手入力**する
   （例: `Microsoft IME` / `Google`）。browser / os は UA から自動判定される。
3. 各試験の前に「クリア」でトレースを空にし、試験後に「JSON エクスポート」で保存する
   （保存先は §7）。ファイル名は `{ime}-{browser}.json` で自動生成される。
4. IME を ON（半角/全角キー等）にし、日本語入力できる状態にする。

## 2. 手順 A — 50 回連続日本語入力（受け入れ基準 #1）

**狙い**: 先頭文字欠落 0 件。移動でフォーカスを別要素へ移さない（I-5）ため、確定→移動直後の
再入力でも先頭 keystroke を取りこぼさないことを確認する。

1. A1 をクリックして選択する。
2. 任意の日本語（例:「あいう」）を入力 → 変換 → **確定 Enter**（このとき下移動しないこと＝#2）
   → もう一度 **Enter で下のセルへ移動**。
3. 2 を **50 回**繰り返す（A1→A2→…と下へ）。各回で「入力した先頭の文字が欠けていないか」を
   目視確認する。
4. **合否**: 50 回で先頭文字欠落が **0 件**なら合格。1 件でも欠落したら、その回のトレース
   （compositionstart 前後・keydown 列）を JSON に残し、不具合として §8 に記録する。

- 記録: 欠落回数 ＿＿ / 50。

## 3. 手順 B — IME 確定 Enter の誤移動（受け入れ基準 #2）

**狙い**: 変換確定の Enter でセルが移動しないこと・その次の独立 Enter で確定＋下移動すること。
R-01 の核心（確定 Enter の発火順 A/B のブラウザー差）を実測する。

1. B2 を選択し、日本語を入力して変換候補を出す。
2. **変換確定のための Enter**（候補確定）を押す。→ **セルが移動しないこと**を確認（誤移動 0）。
3. 続けて **もう一度 Enter** を押す。→ セルが確定し **下へ 1 移動**することを確認。
4. 2〜3 を **10 回**繰り返す。**合否**: 確定 Enter による誤移動が **0 件**なら合格。
5. **観察（重要）**: トレースで確定 Enter の `keydown{Enter}` が `isComposing:true`（順序A）か
   `compositionend` の後で `isComposing:false`（順序B）かを記録する。Chrome/Edge・MS/Google で
   差が出るかを 4 環境で比較する（§8 の比較表へ）。

- 記録: 誤移動回数 ＿＿ / 10、確定 Enter の順序（A/B）＿＿。

## 4. 手順 C — 移動後の再入力 5 経路（受け入れ基準 #3）

**狙い**: 矢印 / Enter / Shift+Enter / Tab / Shift+Tab の各移動直後に日本語入力が成功すること
（成功率 100%）。

1. 任意セルを選択し、次の 5 通りで移動 → 各移動直後に日本語を 1 語入力・変換・確定する。
   - ① ArrowRight で移動 → 入力
   - ② Enter（下）で移動 → 入力
   - ③ Shift+Enter（上）で移動 → 入力
   - ④ Tab（右）で移動 → 入力
   - ⑤ Shift+Tab（左）で移動 → 入力
2. 各経路を **5 回ずつ（計 25 回）**行い、入力開始に失敗（先頭欠落・入力が始まらない）が
   ないか確認する。**合否**: 25 回すべて成功なら合格。

- 記録: 成功回数 ＿＿ / 25（失敗経路: ＿＿）。

## 5. 手順 D — 変換中の Canvas 再描画（受け入れ基準 #4）

**狙い**: 変換中に Canvas が再描画されても変換中文字列（draft）が消えないこと。

1. C3 を選択し、**変換中（未確定・候補ウィンドウが出た状態）**にする。
2. その状態のまま、下部「リモート更新シミュレーター」の **「他セルへ書込」** を数回、続けて
   **「連続書込 開始」**（インターバル再描画）を数秒間有効にする。
3. 変換中の未確定文字列が **消えない / 乱れない**こと、候補ウィンドウが維持されることを確認する。
4. 「連続書込 停止」で止め、そのまま変換を確定できることを確認する。
   **合否**: 変換中文字列の消失が **0 件**なら合格。

- 記録: draft 消失 ＿＿ 件。

## 6. 手順 E — 変換中のリモート更新と競合（受け入れ基準 #5）

**狙い**: 変換中に編集中セル・他セルへリモート更新が来ても、ドラフトが消えず、編集中セルには
競合インジケーターが出ること（MarkConflictOnly・§11.7）。

1. A1 を選択して **変換中**にする（未確定）。
2. **「他セルへ書込」** を押す → 別セルの値が変わり Canvas 再描画されるが、A1 の変換中文字列が
   保持されることを確認（ドラフト消失 0）。
3. 続けて **「編集中セルへ書込（競合）」** を押す → A1 に **競合インジケーター**（編集枠の赤い
   アウトライン／セル角の赤マーカー）が出て、変換中文字列は保持されたままであることを確認。
4. その状態で確定 Enter を押しても、競合が未解決のうちはサイレント上書きされず commit が保留
   される（移動しない）ことを確認（Q-2）。
   **合否**: ドラフト消失 0 件 かつ 競合インジケーター表示ありなら合格。

- 記録: ドラフト消失 ＿＿ 件、競合インジケーター表示（あり / なし）。

## 7. トレースの保存（受け入れ基準 #6）

- 各手順のあと、トレースパネルの **「JSON エクスポート」** で `{ime}-{browser}.json` を保存し、
  `doc/DD/DD-002/traces/phase6-acceptance/` に **手順・環境が分かる名前**へリネームして置く
  （例: `handA-msime-chrome.json` / `handB-google-edge.json`）。
- 不具合時は、その事象を含むトレース（compositionstart/update/end・keydown/keyup・input の列）を
  必ず残す。原因分析は「確定 Enter の順序」「isComposing の値」「input の value」を突き合わせて行う。
