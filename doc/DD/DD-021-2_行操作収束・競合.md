## 目的

行操作を含む同時編集が全クライアントで同一状態へ収束することを保証・実証する: 同一アンカー同時 Insert（サーバー受付順・両方保持=親③）・SetCells×DeleteRows 競合（計画書 §10.3）・reconnect 後収束・randomized 収束スイートの行操作ミックス強化。あわせて K4（IME 編集中の対象行削除）の挙動を synthetic で確定する（ドラフト保持・commit 時 reject=親④/D7）。

## 背景・課題（親 DD-021 §背景の該当分）

- 収束基盤は実装済み: `packages/collab/src/session.ts`（楽観適用・rollback=un-tombstone in place・noop echo 処理=S-E4）・`packages/server-hono/src/convergence.test.ts`・`tests/invariants/collab`（randomized 収束）・reconnect E2E（`apps/playground/e2e/reconnect-headed.spec.ts`）。ただし**利用者発火経路（DD-021-1 の公開 API）を通した**収束・競合 UX（rejected 通知が利用側へ届くか）は未検証。
- 競合マトリクス（計画書 §10.3）の該当行: SetCell×DeleteRow=「DeleteRow が先なら SetCell 拒否（`target-row-deleted`）・SetCell が先なら後から行削除」／InsertRows×同一 anchor=「サーバー受付順・新 RowId で両方保持」。core/validate は実装済み・**公開経路での通知 UX と網羅的シナリオ検証が本子DDの新規**。
- K4（stage2-backlog §3）: IME 編集行の消失検知は `packages/grid/src/ime-editing-session.ts` に synthetic テストあり。「編集継続・ドラフト保持・commit 時 reject 通知」の一貫挙動（§10.4 の精神）としての確定・E2E 化が未了。

## 検討内容

- **収束シナリオ体系**（Phase 0 で一覧化・オーケストレータ確認）: ①同一アンカー同時 Insert×2〜N クライアント ②Insert×Delete 交錯（アンカー行の削除と同時挿入=tombstone 済みアンカー有効・S-D2） ③SetCells×DeleteRows 両順序 ④再 Delete 冪等（S-E4 公開経路） ⑤offline 中の行操作→reconnect catch-up ⑥randomized ミックス（既存スイートの操作比率へ insert/delete を強化・hash 一致・二重適用 0）。
- **通知 UX**: 他者削除行への自分の SetCells reject は既存 rejected（GridConflict）経路で利用側へ届くことを公開 API 経由で検証（新語彙は原則追加しない見込み・必要なら error-codes へ）。
- **K4 挙動（親④/D7 確定）**: リモート DeleteRows は Document State へ適用・編集中ドラフト/textarea は変更しない・編集 UI に行消失を示す・commit 時に `target-row-deleted` で reject 通知・ドラフトは保持（利用者が明示的に破棄するまで消さない）。`ime-editing-session.ts` の既存消失検知を UX として完結させる。**editor-state-machine への遷移追加はしない**（必要と判明したら停止・親ヘッダ条項で xhigh 昇格）。
- **rollback 経路**: 自分の楽観 Insert が reject された場合の巻き戻し（挿入行の除去）と選択・描画の整合（詳細再ベースは DD-021-3・ここではクラッシュしない/収束することまで）。

## 決定事項（親 要確認の確定〔2026-07-17〕の継承）

- **親③**: 同一アンカー同時 Insert は**サーバー受付順・新 RowId で両方保持**。意図順保証なし・rejected にしない。この並び規則を仕様として文書化（本子DDのシナリオ資産＋親の既知制約記録）。
- **親④（IME 分）**: IME 編集中の対象行削除=ドラフト保持・commit 時 reject 通知（強制破棄なし）。数式参照の維持は対象外（DD-022 送り）。
