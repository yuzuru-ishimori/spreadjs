# DD-006 ベンチマーク規約（計測手続き・証跡スキーマ・結論の出し方）

> Phase 0「テスト設計（Red）」の一次成果物。**計測結果を見る前に評価方法を固定する**ための規約。外部レビュー（2026-07-12・第2回）助言 #1「ベンチ規約を結果を見る前に固定」・#3「データ分布を3種類以上」・#5「Worker導入条件を成果物にする」を反映。
> 対象タスク: DD-006 Phase 1（CellStore ベンチ）・Phase 3（recalc ベンチ）・Phase 4（replay/snapshot）・Phase 5（ブラウザ最小確認）。数値の**上限値・関数仕様は `function-spec.md`**、**検証シナリオは `scenarios.md`** が正典で、本書は「どう測り・どう結論づけるか」を定める。
> 記載の回数・倍率は **提案値**。Phase 1 実装前詳細化で確定し、**確定後は結果を見てから変えない**（後付けの基準変更を禁止する）。

---

## 0. 大原則

1. **評価方法を結果より先に凍結する**。ウォームアップ・試行回数・集計指標・GC・実行順・乖離判定を Phase 1 詳細化で確定し、以後は計測値を見て基準を動かさない。
2. **単一の勝者を無理に選ばない**（助言 #2）。CellStore は用途（疎/密・走査/更新）で有利構造が異なりうるため、結論は「用途別選択表」を許容する（§6）。
3. **合否と参考を分ける**。ADR/No-Go判断に効く値（AC1・AC5・AC9 等）と、Worker 分離・snapshot 設計の**素材**（合否対象外）を証跡上で明確に区別する。
4. **全値に再現性を持たせる**。seed・環境・生 JSON を残し、第三者が再実行で追える形にする。

## 1. 計測環境の記録（毎回 JSON へ埋め込む）

- **参照端末**: 本機（DD-004 と同一の Win11・機種/CPU/RAM）。機種情報をレポートへ記録。
- **ランタイム版数**: Node（`process.version`）／Chrome／Edge の各ビルド番号。ブラウザは `navigator.userAgent` と実バージョンを記録。
- **主評価は Node 22（V8）**（要確認3回答）。方式間の**相対比較**が目的。ただし §5 の乖離判定で採用候補は Chrome/Edge 実測（AC9）と突き合わせる。
- **フラグ**: Node は `--expose-gc`（GC 明示のため）で起動。`--max-old-space-size` を既定から変えた場合は記録。

## 2. 計測手続き（提案値・Phase 1 で確定）

| 項目 | 提案 | 意図 |
|------|------|------|
| ウォームアップ回数 | **3回**（計測に含めない） | JIT 最適化・初回アロケーションの除外 |
| 本計測回数 | **10回** | 分布を取るため。全 raw を保存 |
| 主採用指標 | **中央値** | 外れ値に頑健。生成・一括ロードの代表値 |
| 併記指標 | **p95・worst** | ランダム読書き・範囲走査の尾を評価。worst は外れ値監視 |
| GC タイミング | 各試行の計測区間の**直前**に `global.gc()`。区間内は GC を誘発しない（区間を短く保つ）。メモリ計測は計測後に `global.gc()`→安定化してから採取 | GC ノイズの計測区間からの排除（DA #5） |
| 実行順ローテーション | 4方式を試行ごとに巡回（方式 A→B→C→D、次試行 B→C→D→A…）。分布・操作種別ごとに独立ループ | ウォームアップ/GC/キャッシュの順序バイアス排除（DA #5） |
| 再実行 | 全データ・操作列は seed 付き PRNG。**seed を JSON に記録**し同一 seed で再現 | 決定論・第三者再現 |
| メモリ採取 | `process.memoryUsage()`（rss/heapUsed/external）＋`v8.getHeapStatistics()`＋方式別概算フック `approxMemoryBytes()` | 多面計測（単一指標依存を避ける） |

## 3. 生計測 JSON スキーマ（証跡形式・案）

1計測 = 「分布 × 方式 × ランタイム」の1レコード。`DD-006/measurements/` 配下へ保存（例: `cellstore-node-<seed>.json`）。レポート（`measurement-report.md`）はこの生 JSON を集計して作る。

```json
{
  "meta": {
    "seed": 123456,
    "runtime": "node",            // "node" | "chrome" | "edge"
    "runtimeVersion": "v22.x.x",
    "host": { "os": "Windows 11", "cpu": "...", "ramGB": 16 },
    "timestamp": "2026-07-12T00:00:00Z",
    "warmup": 3,
    "trials": 10,
    "acRelevant": true            // 合否対象か（false=参考計測）
  },
  "distribution": "uniform-sparse", // §4 の4種
  "store": "chunked-column",        // map | chunked-column | chunked-rowslot | columnar
  "cellCount": 500000,
  "metrics": {
    "loadMs":       { "median": 0, "p95": 0, "worst": 0, "raw": [] },
    "randomReadMs": { "median": 0, "p95": 0, "worst": 0, "raw": [] },
    "randomWriteMs":{ "median": 0, "p95": 0, "worst": 0, "raw": [] },
    "rangeScanMs":  { "median": 0, "p95": 0, "worst": 0, "raw": [] },
    "memoryBytes":  { "rss": 0, "heapUsed": 0, "external": 0, "approxStore": 0 }
  }
}
```

- `raw` に全試行値を残す（中央値/p95/worst は導出値）。
- recalc（Phase 3）・replay/snapshot（Phase 4）も同型の `meta`＋固有 `metrics` で保存（`meta.acRelevant` で合否/参考を区別）。

## 4. データ分布（4種・助言 #3）

現行の疎/密だけでは実業務との中間が抜けるため、**最低4分布**を同一 seed 系列で生成し全方式に適用する。分布パラメータ（非空率・クラスタリング・型混在比）は可変にする。

| キー | 分布 | 代表する業務像 |
|------|------|----------------|
| `uniform-sparse` | 一様な疎データ（50,000×200・非空500,000） | 全面にまばらな入力 |
| `dense-block` | 連続した密ブロック（矩形500,000） | 高密度の数値領域 |
| `top-left-cluster` | 上部・左側に集中した業務入力型 | 実際の手入力（先頭行・左列に集中） |
| `column-typed` | 列ごとに文字列/数値が偏る | 列＝項目で型が固定される業務表 |

- ストレッチ（2,000,000非空）は各分布で**参考値**として取得（合否対象外・§21）。
- **DA #2 のリスク**（分布で優劣が反転し方式選定が歪む）に直接対応。レポートは分布ごとに結果表を分け、分布前提を明記する。

## 5. Node ↔ ブラウザ乖離の事前判定規則（助言 #1・AC9）

「極端な乖離」を定性判断にしない。採用候補方式を Chrome/Edge 実測（`pocd-browser-bench`）し、Node 実測と次の**事前規則**で突き合わせる。

```text
採用候補方式について、ブラウザ実測 / Node 実測 の比が
  ・処理時間（中央値）で 2.0 倍超、または
  ・ピークメモリで 1.5 倍超
のいずれかに該当する場合、Node 結果をそのまま採用せず、
原因分析（GC・メモリ上限・JIT・DevTools オーバーヘッド等）を必須とし、
方式選定への影響をレポートへ記載する。
```

- 倍率（2.0／1.5）は Phase 1 詳細化で確定。**確定後は結果を見てから変えない**。
- §18.6 No-Go「ブラウザーメモリ上限超過」は Node だけでは判定不可（DA #1）。レポートには **Node/ブラウザの両値を必ず並記**する。

## 6. 結論の出し方（成果物テンプレート・判定は計測後）

計測後、単に数値を残すだけでなく次の**結論表**を埋める。値が入るまでは空欄。

### 6.1 CellStore 用途別選択表（助言 #2・ADR-011 拡充素材）

「チャンク方式を全面採用」に限定せず、用途別の推奨も許容する。

| 用途・条件 | 推奨方式（計測後に記入） | 根拠（生 JSON 参照） |
|------------|--------------------------|----------------------|
| 疎な業務表 | （記入） | |
| 高密度数値領域 | （記入） | |
| 初期 MVP の既定 | （記入） | |
| 再検討条件（非空率・列型・範囲走査頻度） | （記入） | |

### 6.2 Worker 分離判断表（助言 #5・§14.5 素材・合否対象外）

recalc ベンチ（AC2）の影響式数別実測（影響100式以下=合否／1,000・10,000式・10,000行範囲SUM・10,000式チェーン=素材）から、閾値ではなく**判断表**を出す。

```text
影響式数 <= N        → メインスレッド同期再計算
影響式数 > N         → Worker 候補
巨大範囲参照あり     → interval index 必須
深い依存チェーンあり → 分割実行または Worker 必須
```

- N・具体条件は計測後に記入。Worker 分離の実装自体は Phase 1（本DDスコープ外）。

### 6.3 snapshot 閾値の扱い（助言 #6）

replay の素朴 JSON 化計測（serialize/parse 時間・サイズ・復元後メモリ）は**桁感の把握**であり正式 snapshot 形式ではない。したがって結論は—

- ❌「本番 snapshot 間隔を確定」
- ✅「**Phase 1 で正式 snapshot 形式を設計するための暫定推奨値**」

—に留める（§16.3）。正式形式の設計は Phase 1。
