# DD-006 数式仕様（関数・parser資源制限）

> Phase 0「テスト設計（Red）」の一次成果物。`packages/sheet-formula` の parser／evaluator が満たす独自仕様を定義する。
> 正典は計画書 `doc/plan/nanairo_realtime_spreadsheet_development_plan_v1.md` の **§14（数式エンジン）**・**§2.3（数式MVP）**・**§21（性能目標）**。本書は計画書を PoC-D 実装向けに具体化したもので、計画書と矛盾する場合は計画書が優先。
> 対象タスク: DD-006 Phase 2（`tokenizer`/`parser`/`limits`/`bind`）・Phase 3（`evaluator`/`dep-graph`）。上限の具体値は本書を単一の情報源とし、`scenarios.md`・テストコードは本書の値を参照する。

---

## 0. スコープと前提

- 対象文法は計画書 §14.2 MVP文法。**比較演算子（`=` `<` `>` `<=` `>=` `<>`）は文法上予約のみで MVP では拒否**（`IF` 導入時に有効化・§14.2）。
- 実装関数は **`SUM` / `AVERAGE` / `MIN` / `MAX` / `COUNT` の5関数**（§14.7 確定候補・要確認2回答）。`IF`・`AND`・`OR`・`ROUND` 系は追加候補でスコープ外。
- 文法はロケール非依存（§14.8）: 小数点は `.`、引数区切りは `,`。全角数字・`ja-JP` 表示整形はセル入力層の責務で、**数式文字列内では正規化しない**（全角数字を含む式は構文エラー）。
- `$` 絶対参照は **構文解釈のみ**受け付け、AST 属性（`rowMode`/`columnMode`）に保持する。フィル・コピー時の rebind 適用は Phase 1（DD-006 スコープ外）。
- 数式は外部入力であり、parser／evaluator は資源制限（§1）で必ず有界化する（計画書 §20系「evalしない・入力を検証する」の系譜・DD-006 AC8）。

## 1. parser／evaluator 資源制限（AC8 の上限値）

同期評価のためタイムアウトを使えないので、**入力量・構造深さ・処理量を明示上限で打ち切る**。超過時は**例外を投げず対応エラー値を返し**、スタック枯渇・フリーズ・暴走を起こさないことを AC8 で検証する。

| # | 制限項目 | 上限値（提案） | 超過時のエラー値 | 根拠・備考 |
|---|---------|---------------|-----------------|-----------|
| L1 | 数式文字列の最大長 | **8,192** 文字 | `#ERROR!` | Excel の数式長上限と同値。tokenizer 入力量を有界化。`=` を含む全長で判定 |
| L2 | AST 最大ノード数 | **8,192** ノード | `#ERROR!` | parser のメモリと再計算コストを有界化。L1 と整合（1トークン≒1ノード以上） |
| L3 | 括弧・関数呼び出しの合成ネスト最大深さ | **64** | `#ERROR!` | Excel の関数ネスト上限と同値。**再帰下降 parser のスタック枯渇防止（最重要）** |
| L4 | 1関数あたり最大引数数 | **255** | `#ERROR!` | Excel の引数上限と同値 |
| L5 | 単一範囲参照の最大矩形セル数 | **10,000,000** | `#REF!` | アドレス可能空間（50,000行×200列＝§21）と同値。範囲は展開せず interval 保持し、評価は非空セルのみ走査する |
| L6 | 1式あたり evaluate 処理量上限 | **4,000,000** step | `#ERROR!` | タイムアウト代替の処理量カウンタ。基準500,000非空・ストレッチ2,000,000非空（§21）の全域 SUM を1式で処理できる余裕を持ち、その上で暴走を遮断 |

**step の定義（L6）**: セル読み取り1回・算術演算1回・関数引数1件の処理・範囲内セルの走査1件を各 1 step として加算する。上限到達で即時に `#ERROR!` を返し評価を中断する。

> 上限値は「提案」。5値とも AC8 の合否しきい値になるため、実装（Phase 2/3）で実測を得た段階で調整余地あり。調整時は本書と `scenarios.md`・テストを同時更新する（本書が単一の情報源）。Excel 準拠値を初期採用したのは、業務ユーザーの直感と一致し「業務で書きうる式は通り、攻撃的入力は弾く」水準にあるため。

## 2. 値の型と数値変換規則

evaluator が扱う値は計画書 §6.4 `CellScalar` に対応する。5関数・算術演算での型変換を以下に統一する。

| 入力 | 算術演算・関数の**スカラー引数**での扱い | **範囲**内での扱い |
|------|------------------------------------------|---------------------|
| 数値 (`number`) | そのまま数値 | 集計対象 |
| 空白 (`blank`／空文字セル) | 算術文脈では 0 相当。ただし集計関数の**要素数には数えない**（§3参照） | **無視**（集計対象外・要素数に数えない） |
| 文字列 (`string`) | ロケール非依存で数値変換を試みる（`.` 小数のみ）。変換不能なら `#VALUE!` | **無視**（集計対象外） |
| 日付 (`date` ISO) | MVPの5関数では数値化しない → スカラー算術は `#VALUE!` | **無視**（Phase 1 で epoch day 換算を検討・§14.8） |
| エラー値 | そのまま伝播（§4） | 関数により伝播 or 無視（§3・§4） |

- **範囲内の文字列・空白・日付を「無視」**するのは Excel の `SUM`/`AVERAGE`/`MIN`/`MAX`/`COUNT` と同じ挙動で、業務データ（数値列に混在する見出し・空欄）で直感どおりに集計するため。
- **スカラー引数の文字列は数値変換を試みる**（例 `SUM("12", 3)` → 15）。`"12"` のような数値文字列のみ通し、`"abc"` は `#VALUE!`。
- 全角数字・桁区切りは数式内では変換しない（§0）。`SUM("１２")` は `#VALUE!`。

## 3. 関数仕様（5関数）

全関数共通:

- **引数**は「スカラー値」または「範囲参照」を混在可。各関数 **最小1引数**（0引数は `#ERROR!`＝L4 系の引数数検査）。最大引数数は L4（255）。
- **範囲の走査順は行優先（row-major：行→列）** に固定し、「範囲内で最初に出会うエラー」を決定論的にする（§4）。
- 数値変換・空白/文字列/日付の扱いは §2 に従う。

| 関数 | 意味 | 空白 | 文字列 | 日付 | 範囲内エラー | 数値0件のとき |
|------|------|------|--------|------|--------------|----------------|
| `SUM` | 数値の合計 | 無視 | 範囲=無視／スカラー=数値変換 | 無視 | **伝播**（最初のエラー） | `0` |
| `AVERAGE` | 数値の平均（合計÷個数） | 無視（分母に数えない） | 範囲=無視／スカラー=数値変換 | 無視 | **伝播** | **`#DIV/0!`** |
| `MIN` | 数値の最小 | 無視 | 範囲=無視／スカラー=数値変換 | 無視 | **伝播** | `0` |
| `MAX` | 数値の最大 | 無視 | 範囲=無視／スカラー=数値変換 | 無視 | **伝播** | `0` |
| `COUNT` | **数値**の個数 | 数えない | 数えない | 数えない | **無視**（エラーでも結果はエラーにしない） | `0` |

補足:

- **`COUNT` だけがエラーを伝播しない**。COUNT は「数値の個数」を返す目的の関数で、範囲内のエラー・文字列・空白を単に数えないのが独自仕様（Excel の `COUNT` と同旨）。他4関数は範囲・引数にエラー値があれば結果もエラー（Excel の `SUM` 等と同旨）。
- `AVERAGE` の数値0件（全て空白・文字列）→ 0除算になるため **`#DIV/0!`**。
- `MIN`/`MAX` の数値0件 → `0`（Excel と同じ。空範囲を最小/最大 0 とみなす簡略仕様。既知の割り切りとしてレポートに記載）。
- 引数がスカラーで空白セル参照（例 `SUM(A1)` で A1 空）→ 数値0件扱いで各関数の「0件のとき」に従う。

## 4. エラー値と伝播

採用エラー値は **6種**（§2.3・要確認5回答）。エラーは値として伝播する。

| エラー値 | 発生条件 | 検出フェーズ |
|----------|----------|--------------|
| `#ERROR!` | 構文エラー（不正トークン・文法違反・比較演算子）／資源制限 L1・L2・L3・L4・L6 超過 | tokenizer/parser（構文）・evaluator（L6） |
| `#NAME?` | 未知の関数名（5関数以外の呼び出し） | parser／bind |
| `#REF!` | 参照先セル/行/列が削除された／範囲が L5 超過・アドレス範囲外 | bind／evaluate |
| `#CYCLE!` | 循環参照（自己参照・相互参照・範囲経由の循環） | dep-graph（評価前） |
| `#DIV/0!` | 0除算（`/0`・`AVERAGE` の数値0件） | evaluate |
| `#VALUE!` | 型不一致（数値変換不能な文字列・日付のスカラー算術） | evaluate |

**発生フェーズの優先順位**（同一式で複数該当しうる場合、先に確定したものを式の結果とする）:

1. **構文・資源（parse時）**: `#ERROR!`（L1〜L4）— 式が AST にならなければ以降は評価しない。
2. **名前解決・バインド（parse/bind時）**: `#NAME?`（未知関数）／`#REF!`（範囲外・L5）。
3. **循環（グラフ構築時・評価前）**: `#CYCLE!`。
4. **評価時（左→右・範囲は行優先）**: `#REF!`（削除参照）／`#DIV/0!`／`#VALUE!`／`#ERROR!`（L6）。同一評価内では**最初に到達したエラー**を返す。

例:

- `=1/0+FOO()` → `#NAME?`（未知関数は parse/bind 時に確定し、評価時の `#DIV/0!` より先）。
- `=SUM(A1:A3)` で A2 が `#REF!` → `#REF!`（`SUM` はエラー伝播）。
- `=COUNT(A1:A3)` で A2 が `#REF!`、A1=1,A3=2 → `2`（`COUNT` はエラー無視）。

## 5. 評価の決定性（再計算順）

- 差分再計算は dirty 集合からの **topological order**（§14.4）。同順位（依存が並列）のタイブレークは **セルの固定ID（RowId, ColumnId）昇順**で決定論化し、同一 dirty 集合なら常に同一順・同一結果になること（`scenarios.md` dep-graph §で検証）。
- 範囲の走査順は §3 のとおり行優先固定。
- `eval` / `Function` 等の動的コード実行を一切使わない（AC6・lint `no-eval` 相当）。
