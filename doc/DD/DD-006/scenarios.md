## 1. tokenizer（`tokenizer.ts`）— トークン境界・ロケール非依存

- **数値リテラル**: `12`・`12.5`・`.5`・`0` を NUMBER として1トークン化。`12.3.4`（小数点2つ）⛔ `#ERROR!`。指数表記 `1e3` は MVP文法外 → `1`＋識別子 `e3` に分かれ後段で構文エラー ⛔ `#ERROR!`（既知の制約として記録）。
- **文字列リテラル**: `"abc"` を STRING 化。埋め込み引用は二重化 `"a""b"` → 値 `a"b`。閉じない `"abc` ⛔ `#ERROR!`。
- **セル参照**: `A1`・`AA100`・`$A$1`・`A$1`・`$A1` を CELLREF 化（`$` 位置を属性化）。大文字小文字は非区別（`a1` == `A1`）。
- **範囲参照**: `A1:B2` を RANGEREF 化。全列参照 `A:A`・全行参照 `1:1` は MVP文法外 ⛔ `#ERROR!`。
- **関数名・演算子**: `SUM` `+ - * / ^ ( ) , :` を各トークン化。関数名は非区別（`sum` == `SUM`）。
- **空白の無視**: `= 1 +  2 ` や `=SUM( A1 : B2 )` は空白を読み飛ばして同一トークン列。ただしセル参照内の空白 `A 1` は不正 ⛔ `#ERROR!`。
- **ロケール非依存（§14.8）**: 全角数字を含む式 `=１+２`、全角記号を含む式は数式文法では正規化せず ⛔ `#ERROR!`（正規化はセル入力層の責務）。
- **不正文字**: `@` `#`（エラー値リテラル以外）`;` 等の未定義文字 ⛔ `#ERROR!`。

## 2. parser・AST（`parser.ts`・`ast.ts`）— 文法と canonical 化

演算子優先順位・結合（§14.2 文法どおり。**Excel 準拠**で数学慣習と異なる箇所は明記）:

- **乗除 > 加減**: `=1+2*3` → `7`（AST: `(+ 1 (* 2 3))`）。`=(1+2)*3` → `9`。
- **べき乗は左結合**（文法 `power := unary ('^' unary)*`）: `=2^3^2` → `(2^3)^2 = 64`（Excel と同じ。数学慣習の右結合 512 ではない）。
- **単項マイナスはべき乗より先**: `=-2^2` → `(-2)^2 = 4`（Excel と同じ。数学慣習の `-(2^2) = -4` ではない）。
- **単項連鎖**: `=--5` → `5`、`=-+-5` → `5`、`=- - 5`（空白入り）→ `5`。
- **左結合の減算・除算**: `=10-2-3` → `5`、`=10/2/5` → `1`。
- **括弧**: `=((1+2))` → `3`。深いネストは §4 の L3 で別途検証。
- **文字列・数値リテラル式**: `="abc"` → `abc`、`=1.5` → `1.5`。
- **比較演算子は予約のみ・拒否**（§14.2・function-spec §0）: `=1<2`・`=A1=B1`・`=1<>2` ⛔ `#ERROR!`（文法上トークンは認識するが式としては拒否）。
- **関数呼び出し**: `=SUM(1,2,3)` → `6`。`=SUM(A1:A3, MAX(B1:B2), 10)` が入れ子で AST 化。引数0個 `=SUM()` ⛔ `#ERROR!`（最小1引数）。
- **未知関数**: `=FOO(1)` ⛔ `#NAME?`。
- **canonical AST**: 空白違いの `=1+2` と `= 1 + 2 ` は同一 AST。`=SUM(A1:B2)` は安定した正規シリアライズを持つ（同一式→同一表現）。`$` の有無は AST 属性で区別され canonical に反映（`=$A$1+A1` は2参照が別属性）。

## 3. 資源制限（`limits.ts`）— AC8 境界・超過（例外を投げない）

上限値は `function-spec.md` §1（L1〜L6）を参照。各項目「上限ちょうど=通過／上限+1=対応エラー値」を検証し、**いかなる超過入力でも例外送出・スタック枯渇・フリーズ・暴走が起きない**ことを担保する。

- **L1 式長**: 8,192 文字ちょうどの式 → parse 成功。8,193 文字 ⛔ `#ERROR!`（深い解析に入る前に長さで早期打ち切り）。
- **L2 ASTノード数**: `1+1+1+...` を 8,192 ノード相当まで → 成功。超過 ⛔ `#ERROR!`。
- **L3 ネスト深さ**: `((((…))))` を深さ64 → 成功。深さ65 ⛔ `#ERROR!`（**再帰下降のスタック枯渇を起こさず**エラー値で停止＝本項目が AC8 の核心）。関数ネスト `SUM(SUM(SUM(…)))` も合成深さで同様。
- **L4 引数数**: `SUM(1,1,…)` 255個 → 成功。256個 ⛔ `#ERROR!`。
- **L5 範囲セル数**: 矩形10,000,000セルちょうどの範囲 → 受理（展開せず interval 保持）。超過・アドレス範囲外 ⛔ `#REF!`。
- **L6 評価処理量**: 評価 step が 4,000,000 を超える式（巨大範囲の全走査等）⛔ `#ERROR!`（評価を中断）。上限内の重い式（基準500,000非空の全域 SUM 相当）は完走する。

## 4. 固定IDバインド（`bind.ts`）— A1↔BoundCellReference（AC3/4 ユニット）

前提: モック `AxisView`（RowId/ColumnId ⇄ 表示 index）。

- **A1→Bound→A1 往復**: `A1`（row index0→RowId `r0`、col A→ColumnId `c0`）→ `Bound{rowId:r0, columnId:c0, rowMode:relative, columnMode:relative}`。逆変換で `A1` に戻る。
- **絶対参照の属性保持**: `$A$1` → `rowMode:absolute, columnMode:absolute`。`A$1`・`$A1` は片側 absolute。往復で `$` 位置が保存される。
- **行挿入で表示ずれ・論理不変（AC3）**: `A1` を bind 後、参照行の手前に2行挿入（`InsertRows(at index0, count2)`）→ **RowId `r0` は不変**・A1 表示は `A3`（index+2）へ。指す論理セルは同一。
- **参照先削除で `#REF!`（AC4）**: bind 後に参照行 `r0` を削除 → bind 解決が ⛔ `#REF!`。他参照は不変。
- **行移動**: 参照行が移動 → 表示 A1 は新 index を示し RowId は安定（評価値不変）。
- **範囲の追従**: `A1:B2` を bind 後、範囲内側に行挿入 → 範囲が拡張（下端 index が繰り下がる）。範囲上端より手前への挿入 → 範囲全体が下へシフト。

## 5. 依存グラフ・差分再計算（`dep-graph.ts`）— §14.4・2方式

- **単純依存**: `C1==A1+B1`。A1 変更で C1 が dirty・再計算対象。B1 も同様。
- **ダイヤモンド依存**: `D=B+C`・`B=A`・`C=A`。A 変更 → B,C,D が各**1回だけ**再計算（D は B,C 確定後）。二重評価しない。
- **チェーン**: `E←D←C←B←A` の長鎖（10,000段）。A 変更 → topological 順で全段再計算。
- **範囲依存**: `S==SUM(A1:A100)`。範囲内の任意 `A_k` 変更で S が dirty。**全展開方式・列別interval index方式の両方**が同じ dependents 集合を返す（等価性）。
- **範囲重なり**: `S1=SUM(A1:A50)`・`S2=SUM(A25:A75)`。A30 変更 → S1,S2 とも dirty。
- **循環検出（DFS coloring）**: 相互 `A=B`・`B=A` → 両方 ⛔ `#CYCLE!`。自己 `A=A+1` ⛔ `#CYCLE!`。範囲経由の自己包含 `A1=SUM(A1:A3)` ⛔ `#CYCLE!`。
- **循環解消後の復帰**: 循環辺を除去する変更後、当該セルが再 dirty 化され正常値に再計算される（`#CYCLE!` が残らない）。
- **再計算順の決定性（function-spec §5）**: 同一 dirty 集合に対し、同順位のタイブレークを固定ID昇順にして**常に同一順・同一結果**。2方式間でも最終評価値が一致。

## 6. 評価器（`evaluator.ts`）— 5関数・型・エラー伝播（AC6）

算術（§2 の結果を評価値で再確認）に加え、`function-spec.md` §2〜§4 を検証:

- **SUM**: `SUM(A1:A3)` で A1=1,A2=空,A3="x" → `1`（空白・文字列は無視）。`SUM("12",3)` → `15`（スカラー文字列は数値変換）。`SUM("x",3)` ⛔ `#VALUE!`。数値0件 → `0`。
- **AVERAGE**: A1=2,A2=4,A3=空 の `AVERAGE(A1:A3)` → `3`（空白は分母に数えない）。全て空/文字列 ⛔ `#DIV/0!`。
- **MIN/MAX**: 空白・文字列を無視して最小/最大。数値0件 → `0`。
- **COUNT**: `COUNT(A1:A4)` で A1=1,A2="x",A3=空,A4=`#REF!` → `1`（**数値のみ計数・エラーも無視**）。
- **範囲エラー伝播（COUNT以外）**: `SUM(A1:A3)` で A2=`#REF!` → `#REF!`。`MAX(A1:A2)` で A1=`#DIV/0!` → `#DIV/0!`。
- **算術エラー**: `=1/0` ⛔ `#DIV/0!`。`=A1/A2`（A2=0）⛔ `#DIV/0!`。`="a"+1` ⛔ `#VALUE!`。
- **特殊値（function-spec §2.1）**: 非有限化する式（オーバーフロー例 `=1e308*10`）⛔ `#VALUE!`（`Infinity` を値化しない）。`=0/0` ⛔ `#DIV/0!`（`NaN` を経由しない）。負の0: `=-0` と `=0*-1` の評価値は `0` と同一（`-0` を出さない・`SUM(-0,0)`→`0`）。
- **ロケール不変（function-spec §2.2）**: 同一式・同一データの評価値が実行環境ロケールに依存しない（`.` 小数固定・`toLocaleString` 不使用）。
- **エラー発生フェーズの優先**（function-spec §4）: `=1/0+FOO()` → `#NAME?`（parse/bind が eval より先）。`=COUNT(A1:A3)`（A2=`#REF!`, A1=1,A3=2）→ `2`。
- **動的コード不使用（AC6）**: 実装に `eval`/`Function` を含まない（lint `no-eval` 相当）。評価はインタプリタで行い、`=1+1` の評価が JS `eval` を呼ばない。
- **env-free 純度（助言 #7・AC8隣接）**: `packages/sheet-formula` の実装ファイル（`src/**/*.ts`・テスト除外）を `typecheck:core`（`tsconfig.core.json`・`types:[]`・DOM/Node lib なし）で型検査 → `process`/`window`/`document` 等の環境 API 参照が**型エラーになる**（誤混入を検出）。DD-005 の probe 手法（環境 API を一時挿入して core=検出・main=素通りを実測）で実効性を確認する。

## 7. CellStore候補（`apps/pocd-bench/src/stores/*.ts`）— 4実装3カテゴリ（AC1素材）

共通 `CellStoreCandidate` インターフェイスの4実装（①Map型＝単一Map／②チャンク型2実装＝列チャンクMap・DD-004行スロット移植／③列指向配列型）を対象。

- **等価性（最重要）**: 同一操作列を4実装へ適用 → 全 `get(row,col)` 結果と `nonEmptyCount()` が4実装で一致（実装差はメモリ・速度のみ）。**4分布すべて**（`bench-protocol.md` §4）で等価性が成立する。
- **チャンク境界**: 256行スロット前提で row=255 と row=256（境界跨ぎ）へ set → 双方正しく取得・チャンクが2つ確保される。
- **空行・疎データ**: 大きなギャップを持つ疎分布 → 非空チャンクだけ確保（メモリが非空セル数に比例・全行分の確保をしない）。
- **密データ**: 連続矩形ブロックで列指向配列型が効率的に格納。疎/密どちらでも等価性が成立。
- **業務集中型・列型偏り**: 上部・左側集中（`top-left-cluster`）・列ごとの型偏り（`column-typed`）でも等価性が成立し、方式ごとのメモリ・走査時間が分布別に記録される（分布で優劣が反転しうるため＝DA #2）。
- **可視範囲走査 O(可視非空数)**: `queryRange` が範囲内の非空セルのみ visit（範囲外を1件も visit しない・DD-004 chunk-store と同基準）。全セル空の窓は visit 0回。
- **削除**: 空文字/空へ set で削除扱い（`nonEmptyCount` 減少）。
- **メモリ概算フック**: `approxMemoryBytes()>0`（レポート素材）。

## 8. 決定論データ生成（`data-gen.ts`・`op-gen.ts`）

- **data-gen 再現性**: 同一 seed で完全一致の (row,col,value) 列。**4分布**（`bench-protocol.md` §4）を生成 — `uniform-sparse`（50,000×200・非空ちょうど500,000・全て範囲内・重複なし）／`dense-block`（連続ブロック500,000）／`top-left-cluster`（上部・左側集中の業務入力型）／`column-typed`（列ごとに文字列/数値が偏る）。ストレッチ（2,000,000）は各分布で参考生成。数値・文字列・日付が混在（種別2種以上）。出力は (row,col) 昇順。分布パラメータ（非空率・クラスタリング・型混在比）は可変。
- **op-gen 再現性**: 同一 seed で100,000 Operation列（SetCells/InsertRows/DeleteRows）を再現。混在比率が決定論的（DD-003 fuzzer 踏襲）。生成 Operation は sheet-core が適用可能な妥当形。

## 9. replay・snapshot参考（`bench-replay.ts` 健全性）— AC5素材

計測値そのものは JSON/レポートで担保。vitest では健全性のみ:

- **hash 決定性**: 同一 Operation 列を `sheet-core` の `apply` で2回 replay → 最終 document hash が一致（DD-003 の収束結果と整合）。
- **計測点**: 1,000/5,000/10,000/50,000/100,000 点で所要時間・メモリを記録する関数が JSON を出力する（値の妥当性はレポート判断）。
- **snapshot 参考**: replay 後文書を素朴に JSON 化 → serialize/parse が往復し、復元文書の hash が元と一致（サイズ・時間はレポート素材・合否対象外）。

## 10. sheet-core 結合試験（`integration-sheetcore.test.ts`）— AC3/4 実文書版

`sheet-core` は**読み取り＋`apply` 利用のみ**（既存 package 無変更。Axis 情報が未公開なら読み取り専用の薄いアダプタ）。

- **行挿入で固定ID維持（AC3実文書）**: sheet-core で文書作成 → 数式を RowId/ColumnId へ bind → `InsertRows` を apply → A1 表示は移動後位置・**評価値は不変**。
- **参照先削除で `#REF!`（AC4実文書）**: 同文書で参照行を `DeleteRows` → 該当式が `#REF!`・他式は正常。
- **数式込み replay 参考**: formula 付き SetCells → 一括再計算が sheet-core 文書上で成立（参考計測）。
