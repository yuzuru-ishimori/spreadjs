## 目的

BI・帳票系アプリの「明細レコード閲覧」を SDK の1構成で成立させる。**表示専用モード（readOnly）・列見出しキャプション・数値/日付表示書式**の3点により、「データと列スキーマを渡すだけで業務の表として読める」状態を提供する（既存の columnWidths / wrapColumns / columnTypes / columnFormats に連なる列スキーマの完成）。

## 背景・課題

- 駆動要因: 実案件（BIツール開発）で**明細レコードの閲覧ニーズが高い**ことを確認（2026-07-21・ユーザー）。「Excelにデータ表示したい」の実体は (a) Web 画面での Excel ライク閲覧 と (b) xlsx 持ち出し。本DDは (a) を提供する。(b) Excel I/O は計画書 §2.5 の対象外のまま（範囲コピー TSV→Excel 貼り付け=DD-020 提供済みが代替。スタイル付き xlsx 生成はグリッド外の別コンポーネント候補として、要求が確定したら別DD）。
- 既存実装の実調査（2026-07-21・起票時）:
  - **公開面（先例あり）**: `packages/grid/src/index.ts` の `GridCommonMountOptions` に列単位設定の運用が確立済み（columnWidths/rowHeights=DD-012-4・wrapColumns=DD-012-5・columnTypes=DD-027-1/2・columnFormats=DD-027-3）。mount 時固定・両モード共通・不正設定は mount fail-fast（code=`column-types-invalid`）。
  - **列ヘッダー（キャプション未実装）**: `packages/render/src/base-layer.ts` の `columnLabel()` が A/B/C… を固定描画するのみ（129・464 行）。業務名を出す機構は無い。列ヘッダー自体は Canvas チュームとして常時固定描画＝BI 要件「ヘッダー行固定」は既存構造で充足済み。
  - **数値・日付表示（書式未実装）**: `packages/grid/src/document-view.ts` の `cellScalarToDisplay` による素の文字列化のみ。千位区切り・小数桁・%・日付パターンは無い。
  - **表示専用（未実装）**: `document-view.ts` の内部イミュータブルガード（set/bulkLoad throw）のみで、製品機能としての readOnly は無い。文書変更の入口は `mount-controller.ts` の pointerdown/dblclick/キー入力・paste/cut（DD-020）・行操作ショートカット（DD-021）・Undo/Redo。確定系 chokepoint は `submitSetCells`（DD-020）＋行操作 submit（DD-021）。
  - **契約面**: 表示文字列は cell-commit/setData round-trip・コピー TSV・columnFormats 完全一致（DD-027 決定⑥）が依存する契約面。表示書式がこれを変えるかは要確認⑤。
- 位置づけ: Stage 2 ロードマップ（`doc/plan/phase2-dd-roadmap.md` §1）の採番 DD-023〜032 に無い consumer 駆動の新規要望。roadmap 注記は「想定外DDは子DDで起票」とするが、本件は既存DDの子として不自然な独立機能のため新規トップレベル DD-033 で起票し、ロードマップ上の扱いを要確認①でユーザー確定する。

### D2〜D7（要確認①〜⑥の根拠）

- **D2（①位置づけ）**: S2-1 は既定 consumer 2件で充足予定・roadmap §4 R5 は「3件目 candidate 探索はスコープ外（推測で計画しない）」だが、本件は推測でなく確認済みの実案件ニーズ。**機能提供（本DD）と consumer 統合（BI案件への組み込み）を分離**すれば R5 と衝突せずに前倒しできる。
- **D3（②抑止の層）**: 入口抑止（編集 UI を開かない）だけでは paste・ショートカット等の取りこぼしが構造的に残り、chokepoint 抑止だけでは「編集できたのに確定で失敗する」不自然 UX になる。**両層**（入口=UX・chokepoint=保証）で防ぐ。editor-state-machine 無改変（mount-controller 側の分岐のみ）が設計目標。
- **D4（③キャプション）**: A/B/C の置換描画が最小実装。行番号ヘッダーの非表示化・2段ヘッダーは要求未確認のため対象外。
- **D5（④書式語彙）**: 自前ミニ書式（決定的・依存ゼロ・テスト容易）／Intl 委譲（実装小・ロケール/環境差）／Excel 書式文字列（互換期待の無限拡大=リスク R-15）の3案から、既定=自前ミニ書式。
- **D6（⑤適用面）**: 表示文字列は契約面（背景・課題）のため、書式は **Canvas 描画テキストへのみ**適用が安全。コピーが raw のままなら Excel 貼り付け後に Excel 側で数値書式を適用できる利点もある。columnFormats（DD-027-3）の完全一致も raw 対象のまま＝既存挙動不変。
- **D7（⑥mount 契約）**: wrapColumns/columnTypes の確立済み運用（mount 時固定・両モード共通・fail-fast）を踏襲し、新概念を増やさない。

> **2026-07-21 実行開始時に①〜⑥すべて既定案どおり最終確定**（決定事項参照）。①の記録先のみ DD-032 未起票のため roadmap §4 R5 追記へ変更。

1. **要確認①: ロードマップ上の位置づけと着手順** — 既定案: 新規トップレベル **DD-033 として Stage 2 に追加**（`phase2-dd-roadmap.md` §1 への追記=更新記録）。着手は **DD-027 完了後**（base-layer・mount オプション・showcase を共に触るため）。BI 案件への consumer 統合は本DD対象外とし、3件目 candidate として DD-032 の Stage 3 バックログ棚卸しへ記録（R5 と非衝突）。対案: Stage 3 送り（BI 案件の時期に間に合わないリスク）。
2. **要確認②: readOnly の抑止範囲** — 既定案: 文書変更系を全抑止（編集開始〔キー入力・F2・ダブルクリック・IME〕・paste/cut・Delete/Backspace クリア・行挿入削除〔ショートカット・公開 API〕・Undo/Redo）。閲覧系は維持（範囲選択・コピー・スクロール・列幅行高リサイズ・link-open）。選択式列のドロップダウンは開かない。共同編集モードでは**受信反映のみ**（自クライアントから文書 Operation 送信ゼロ）。`setData`（プログラム的注入）は**許可**（閲覧データの差し替え手段）。抑止時はサイレントにせず onDiagnostic へ notice を出す。**注意: readOnly は権限制御ではない**（サーバー側強制なし・利用側責務として既知制約へ記録）。
3. **要確認③: 列見出しキャプション** — 既定案: `columnCaptions`（ColumnId 文字列→表示名）。指定列は A/B/C を置換描画・未指定列は従来どおり。行番号ヘッダーは現状維持・2段ヘッダーは対象外。
4. **要確認④: 数値・日付書式の語彙** — 既定案: 自前ミニ書式 `columnDisplayFormats`（ColumnId→書式）。number=`{ grouping?, decimals?, percent?, prefix?, suffix? }`・date=固定トークン（`YYYY`/`MM`/`DD`/`HH`/`mm`/`ss`）のパターン文字列。Excel 書式文字列互換・Intl ロケール書式は v1 対象外（拡張点メモのみ）。
5. **要確認⑤: 書式の適用面** — 既定案: **Canvas 描画テキストのみ**整形。cell-commit/setData round-trip・コピー TSV・columnFormats 完全一致・編集ドラフト文字列は従来の表示文字列（raw）のまま＝**契約不変**。
6. **要確認⑥: mount 契約** — 既定案: 3オプションとも両モード共通・mount 時固定（wrapColumns/columnTypes と同運用）。不正設定（未知列・不正パターン・decimals 範囲外等）は mount 時 fail-fast（phase=config・新 error code は子DDで確定〔`column-types-invalid` と同経路〕）。共同編集モードでの全クライアント設定一致は利用側責務。

## 既知制約（Phase 3 記録・公開 JSDoc にも明記済み）

- **readOnly は権限制御ではない**: サーバー側強制の無い view 側の表示制御。悪意あるクライアントは抑止を迂回して Operation を送信しうる（サーバーは readOnly を知らない）。アクセス制御・認可は利用側責務（DD-026 責務境界と同型）。
- **書式・キャプションは view-local**: 文書状態を変えない描画整形のため、共同編集モードで設定不一致のクライアントは異なる見た目を見る（文書 hash は乖離しない）。全クライアントの設定一致は利用側責務（columnTypes/columnFormats=DD-027 と同運用）。
- **「見た目と一致条件のズレ」**: columnFormats の match・選択式候補・編集ドラフト・コピー TSV は raw のまま（契約不変=要確認⑤）。書式済み文字列（例 `1,234`）でこれらは一致しない。
- **date 書式の受理形は ISO 2形のみ**（v1）: 非受理 raw（`YYYY/MM/DD`・和暦・数値シリアル等）は書式適用されず raw のまま表示される（セル単位診断なし）。
- **date 書式はカレンダー妥当性を検証しない**: `2026-13-45` も形式一致すれば `2026/13/45` と整形される（フィールド直取り＝決定的動作の仕様。統合レビュー P3-4）。
- **E2E 観測用 debug API（cellRenderText/columnHeaderText）は描画実体でなく compiledDisplay の再適用**: フック束縛自体の欠陥は検出できない（ピクセル走査 E2E が実体側を補強。統合レビュー P3-2）。
