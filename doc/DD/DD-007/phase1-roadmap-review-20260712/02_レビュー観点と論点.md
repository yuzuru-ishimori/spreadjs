# レビュー観点と論点（自己完結の背景）

> ChatGPT はリポジトリを読めないため、このファイルだけで背景が分かるようにしてある。詳細は `03`（計画本体）・`04`（Phase 0 成果）・`05`（計画書）・`06`（密度レビュー）を参照。

## 1. プロジェクトと体制

- **プロダクト**: 業務Webアプリへ組み込む TypeScript 製のリアルタイム共同編集スプレッドシート型入力基盤。日本語IME・数万行の Canvas 描画・サーバー主導の共同編集を優先。React は業務UIの外周のみで、グリッドコアは依存ゼロ。
- **体制**: 開発者ほぼ1名 ＋ AIエージェント。設計書（DD）の起票・実装・コードレビューをAIが分担。**希少資源は人間の確認時間とAPI/サブスク枠**。「人を増やす」は選べない。
- **開発単位**: DD（設計書）。原則「1 DD = 1レビューサイクル」。各DDは「起票 → 仕様確認ゲート → 実装 → レビュー（DA自己批判＋Codexコードレビュー＋必要に応じ外部レビュー＋実機ゲート）」を通る。
- **3層構造**: ①ロードマップ（DD間の依存・順序・マイルストーン）②個別DD（1レビューサイクル）③ADR・仕様書（長期維持の設計判断）。

## 2. 今どこにいるか

- **Phase 0（技術PoC）＝製品成熟 Stage 0** をほぼ完了。DD-001〜006 で IME／共同編集収束／Canvas 5万行描画／統合フロー／データ・簡易数式の成立性を確認済み（詳細と「確認できていないこと」は `04`）。
- **DD-007 で Phase 0 の Go/No-Go は判定済み＝条件付きGo**（2026-07-12・ユーザー判定）。IME実機検証を筆頭に **CG-1〜6** を解除条件に付す（詳細は `04` §2.5）。
- 本レビュー対象の草案は、その条件付きGo の先の Phase 1 をどう進めるかを設計したもの。→ **草案が CG-1〜6 を各DDへ適切に反映しているか**も観点に含む（論点E）。

## 3. 到達目標＝製品成熟 Stage 1「社内SDK Alpha」

- 開発Phase と製品成熟段階は別軸で管理している（憲章）。**Phase 1 の到達目標は Stage 1 SDK Alpha ＝「別の社内プロジェクトから npm パッケージとして利用可能」**（Goal 1）。詳細な移行条件は `04`。
- つまり Phase 1 は単なる機能実装ではなく、**公開 Facade・consumer fixture・Quick Start・型定義まで揃えて「他アプリから組み込める」ことを完了条件**にする。ただし API は `0.x`（変更可）で、Stable 固定はもっと後段。

## 4. 特に見てほしい論点（草案の要）

### 論点A: フェーズ境界の整合（最重要）

- 開発計画書 §19 では **Phase 1＝単一利用者コア**、**Phase 2＝共同編集Alpha・永続化**と分かれている（`05`）。
- 一方、縦切りロードマップの**最初のDDは確定事項として「日本語でセルを連続入力し、確定値が共同編集で保存される」**（＝単一利用者と共同編集をまたぐ）。
- **直前の密度レビュー（`06` §1.4/§5.1）が「この境界を確定しないと Phase 1 の粒度・密度を決められない」と指摘**していた。
- 草案の回答: **縦切り目標は保ちつつ、最初の縦切りを DD-010（単一利用者コア・保存なし）と DD-011（共同編集保存）に分割**し、§19 の技術境界を DD レベルで保存した。→ **この整合のさせ方は健全か? Phase 2 相当リスクを Phase 1 の軽量枠で扱う穴は残っていないか?**

### 論点B: 縦切りDDの切り方・過積載リスク

- 前回 DD-005（統合PoC）が「1つのDDに詰め込みすぎて重くなった」失敗があった（IME＋Canvas＋共同編集＋パッケージ抽出を1本に）。
- 草案は「1DD＝一つの利用者成果＋一つの支配的リスク」を原則にし、最初の縦切りを3本（DD-010/011/012）に割った。→ **過積載の再発兆候はないか? 逆に細切れで縦切りの意味を失っている箇所はないか?**

### 論点C: Stage 1 Alpha の最小到達ライン

- どこまでを「Alpha成立」とするか。特に **DD-015（切断・再接続で入力を失わない堅牢化＝DD-005 の既知制約回収）を Alpha 必須にするか、強く推奨に留めるか**。
- **DD-016（行追加・削除）・DD-017（最小数式）を Stage 1 に含めるか Stage 2 へ送るか**。

### 論点D: 密度レジーム

- 草案は密度レビュー（`06`）の推奨を Phase 1 プロセス標準として織り込んだ: Risk Class A/B/C をDDヘッダ必須／人間ゲートを例外ベース（B/C は承認済みバックログ内なら自動開始）／Codex effort をリスク別／証跡を5点へ圧縮／実機ゲートを変更トリガー方式／不変条件は常設スイート化。
- → **Phase 1 で実際に機能するか。軽量化しすぎて危険な箇所（データ損失・IME・公開API）／逆に重いままでよい箇所は?**

### 論点E: 条件付きGo 条件（CG-1〜6）の反映

- DD-007 の判定＝条件付きGo。解除条件 CG-1〜6（`04` §2.5）を Phase 1 で回収する必要がある。
- 草案は CG を各DDへ割り付けている（CG-1→DD-010・CG-5→DD-015 等）。→ **割付は妥当か? 特に CG-1 実機IME検証が「Alpha 配布可否の筆頭ゲート」として十分に強く扱われているか、CG-5 のデータ損失境界が Alpha 最小ライン（論点C）と整合しているか。抜けている条件はないか。**

## 5. 品質を落とせない領域（軽量化しても守る）

- IME 正しさ（実機・順序A/B・composition中のdraft不変）。
- 共同編集のデータ整合（全順序hash一致・rollback/replay収束・サイレント上書き禁止・reject時の入力保持・idempotency・reconnect）。
- 公開API契約（Stable/Experimental/Internal 区分・破壊的変更検出・移行ガイド）。
- 性能回帰（該当経路を変えたときのフル再計測）。
