## 目的

利用者成果=「**5万行×200列で scroll/selection が快適（回帰予算内）**」を担保する。PoC-B 描画資産を `@nanairo-sheet/render` へ実抽出（挙動保存）し、§23 Phase 1完了条件「5万行の基本 scroll/selection」を**統合性能回帰ゲート**（合格ライン=DD-004 実測の予算化）として常設化（`tests/invariants/perf`）し、**CG-6（精密メモリ計測）を解除**する。

## 背景・課題

- DD-009 資産台帳の割当: `pocb/{base-layer,overlay-layer,viewport,scroll-anchor,dpi,axis,text-cache,render-scheduler}.ts`＝Harden（→`@nanairo-sheet/render`）。`document-view.ts` 結線（DD-012-1 で grid 側に位置づけ済）を維持。
- **CG-6**: `performance.memory` 封鎖を回避した精密メモリ計測が未実施。期限=Alpha exit前・未解除は「データ上限明示 or Alpha不可」。担当=本DD（roadmap §0）。
- 計測ハーネスの再判断（台帳 Q2/Q3/Q4）: `apps/pocd-bench`（CLI 計測）・`apps/pocd-browser-bench`（500k セル browser 実測・`performance.memory`）・`integration/initial-load-metrics.ts`／`pocb/metrics.ts` は流用を第一候補とし、Phase 2 冒頭で流用/作り直しを判断。
- DD-011 設置の `tests/invariants/perf` は暫定予算の最小スモークのみ → 本DDで実予算（DD-004 実測基準）へ差し替え・常設化。
- boundary baseline のうち `pocb` 由来の R1 entries（owner=DD-012 分）を本DDで縮退。
- **依存: DD-012-1 完了後**（入力経路・selection 抽出が確定した状態の統合ページで計測する。描画抽出は挙動保存のため DD-012-1 の CG-1 実機証拠を無効化しない前提。挙動差が出た場合は実機スモーク再判定を親DDへエスカレーション）。

## 検討内容

- **合格ライン（ユーザー確定③）**: DD-004 実機実測を回帰予算の基準にする — **scroll p95 16.8ms・選択 16.9ms・再描画 0.33ms・メモリ 300MB内**（DD-006 実測整合）。計測ノイズ許容（マージン）の与え方は Phase 2 の実装前詳細化で予算表として固定し、DD本文に記録する（予算を緩める変更は spec 変更＝再ゲート対象）。
- **計測条件**: 5万行×200列（A-03 アドレス可能範囲）。セル充填率・列幅・行高は DD-004 計測条件を踏襲し、差分があれば予算表に明記。
- **CG-6 の計測方式**: `performance.memory` は既定で precision が封鎖されるため、精密値の取得方法（Chromium 起動フラグ等）＋`pocd-browser-bench` の流用可否を Phase 3 冒頭で確定。単発値でなく時系列（リーク傾向）も記録（DD-004 の手法踏襲）。

## 決定事項

- 抽出は**挙動保存**（描画結果・キャッシュ挙動・DPI 挙動を変えない）＋既存テスト green 維持。規約・設計の新規発明をしない。
- 性能予算は `tests/invariants/perf` に**常設回帰ゲート**として固定し、以後のDD（DD-013〜016）が §2.3 L4 の発動条件（描画キャッシュ・replay 方式等の変更）でフル再計測する基盤にする。
- 5万行 headed 実測・メモリ計測は再現コマンド付きでレポート化（`doc/DD/DD-012-2/`）。
- 新 workspace（render）追加は package-lock 変更を伴う → 着手前に `git status`＋並行セッション確認。
