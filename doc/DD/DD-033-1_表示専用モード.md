## 目的

`readOnly: true` の mount オプション1つで「文書を一切変更できない閲覧専用グリッド」を提供する（親DD-033 の第1子＝明細閲覧ビューのモード基盤）。閲覧系操作（選択・コピー・スクロール・リサイズ・link-open）は全て維持する。

## 背景・課題（親確定値の転記・2026-07-21 ユーザー最終確定）

- **要確認②（抑止範囲）**: 文書変更系を全抑止 — 編集開始（キー入力・F2・ダブルクリック・IME）・paste/cut・Delete/Backspace クリア・行挿入削除（ショートカット・公開 API）・Undo/Redo。閲覧系は維持 — 範囲選択・コピー・スクロール・列幅行高リサイズ・link-open。選択式列のドロップダウンは開かない。共同編集モードは受信反映のみ（自クライアントから文書 Operation 送信ゼロ）。`setData` は許可（閲覧データの差し替え手段）。抑止時は onDiagnostic へ notice。**readOnly は権限制御ではない**（サーバー側強制なし・利用側責務。既知制約への記録は親 Phase 3）。
- **要確認⑥（mount 契約）**: readOnly は両モード共通・mount 時固定（wrapColumns/columnTypes と同運用）。不正設定の扱い（fail-fast 要否・新 error code）は本子DDの📐で確定（→ 決定事項）。共同編集モードでの全クライアント設定一致は利用側責務。

## 検討内容

- **2層抑止の設計**（親D3の具体化）: 入口抑止だけでは synthetic composition・将来追加される編集入口の取りこぼしが構造的に残り、chokepoint だけでは「編集 UI が開くのに確定できない」不自然 UX になる。入口=UX（編集 UI を一切開かない）・chokepoint=保証（文書 Operation 送信ゼロ）の両層とする。
- **textarea.readOnly 属性＋アダプタ層 dispatch 抑止の併用**: 属性だけでは synthetic composition イベント（dispatchEvent）が state machine へ届き BeginEdit が起きる（E2E で編集 UI 非表示を検証できない）。dispatch 抑止だけでは実 IME の候補ウィンドウがユーザーに見えてしまう。両方で「実機でも synthetic でも編集 UI が開かない」を成立させる。readOnly=false 時は integration-editor は完全無変更（分岐追加のみ）。
- **入口裁定の純関数化**: `readonly-policy.ts`（新規・DOM 非依存）に「このキー/操作は readOnly で抑止するか」の裁定を集約し TDD 対象とする（`decideNavigationIntercept` 等の既存 decide* 流儀）。矢印・Shift+矢印・Ctrl+C・Escape・PageUp/Down 等の閲覧系は pass。

## 決定事項

- 親②⑥の確定値（背景に転記）＋📐 の設計（editor-state-machine/session-sync/standalone-session 無改変・入口＋chokepoint 2層・textarea readOnly 属性＋dispatch 抑止併用・純関数裁定）を採用。
- **fail-fast 不要・新 error code 追加なし（要確認⑥の子DD確定分・📐）**: readOnly は boolean 単値で、columnTypes のような構成不整合（未知列・重複・併用矛盾）が構造的に存在しない。TS 型（`boolean | undefined`）が正規経路を強制する。JS 経路の非 boolean 値は `readOnly === true` の厳密判定で編集可能側へ倒し、`undefined` 以外の非 boolean は mount 時に診断 warn（診断 code=`readonly-invalid`・公開 GRID_ERROR_CODES へは追加しない）。理由: fail-fast に値する破壊的誤設定が無く、公開エラー語彙の増設コスト＞益。
- **抑止時の診断**: code=`readonly-blocked`・level=`info`（onDiagnostic へ notice 相当・opt-in ゆえ既定無コスト）。mount 時に readOnly=true なら `readonly-mode` info を1件出す（E2E/障害切り分けの確認点）。
- 公開 API/イベントの追加は `readOnly` オプション1点のみ（新イベント・新公開エラーコードなし）。CHANGELOG・API 型 snapshot 記録は親 Phase 3（AC8）。
- **要確認1〜4はすべて既定案どおり確定（ユーザー確定 2026-07-21・親セッション仕様確認ゲート）**: ①T1 実施（親 Phase 3 受付・Claude 代行可） ②親AC3 は E2E（collab）＋debug API＋裁定純関数 unit で検証（親AC3 の検証方法欄も同旨へ更新済み） ③presence 送信は維持（閲覧位置共有は閲覧系） ④列境界 dblclick の auto-fit は維持（view-local layout＝閲覧系）。
