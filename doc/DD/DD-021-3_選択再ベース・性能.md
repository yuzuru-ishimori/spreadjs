## 目的

K3（行挿入/削除後のローカル選択・Enter 移動先の再ベース）を回収し、行構造変更下でも activeCell・選択レンジ・移動先が**行実体（RowId）に追従**するようにする。あわせて Undo スタックとの整合（削除行に触れるエントリの実行時拒否・除去＋通知=親①）と、P2-1（単一行 InsertRows 連発の Θ(N²)・`apply.ts`）の性能是正を行う。

## 背景・課題（親 DD-021 §背景の該当分）

- **K3 の実体**: `packages/ime/src/editor-state-machine.ts` の `activeCell` と `packages/grid/src/selection-controller.ts` の選択レンジ（anchor/focus）はいずれも**表示 index ベース**（`{row, col}`）。リモート/ローカルの行挿入・削除で `document-view.ts` が rowAxis を再構築しても index は補正されず、カーソル下の行実体がずれる（挿入が上で起きると 1 行下の別行を編集してしまう）。Enter 確定の移動先も同様。
- **Undo 整合**: `packages/grid/src/undo-stack.ts`（DD-020-3）の補償 SetCells は削除行に対し `target-row-deleted` で reject されるが、エントリ処理（除去・通知）と挿入行下での beforeRevision 妥当性が行操作を前提に検証されていない。行操作自体の Undo は対象外（親①=計画書 §15.3 MVP後）。
- **P2-1 の実体**: `packages/core/src/apply.ts` の `nextSlot`（rowMeta **全走査** O(N)）が Insert 1 回ごとに走り、`rowOrder.splice`（O(N)）と合わせ単一行 Insert 連発ログで Θ(N²)（DD-014 既知制約。bulk insert の O(N²) 回避=`replayAcceptedOperations` は実証済み・単発連発が未回収）。replay・rollback 経路と同一関数を共有するため、是正は決定論（同一入力→同一文書）を壊さないことが必須。

## 検討内容

- **再ベース方式（📐 で確定）**: 案(a) editor-state-machine の位置表現を RowId 化＝状態機械の内部変更（大改修・xhigh 昇格トリガー）。案(b)〔既定〕**grid 層の再ベース hook**＝row-structure 変更（`operationDirtyKind === 'row-structure'`）を検知した時点で「変更前に指していた RowId」を `document-view` の新 rowAxis で index へ引き直し、既存 API（状態機械の位置設定・selection-controller の setRange）で補正する（**遷移追加なし・最小侵襲**）。追従用の「現在指している RowId」は grid 層（mount-controller）が所有する。
- **削除時の縮退（親④確定）**: activeCell の行が削除されたら最近傍生存行（下優先→無ければ上）へ移動。選択レンジは生存行へ縮退（全行消失なら単一セルへ）。DD-021-1 のローカル削除縮退を**リモート起因へ一般化**し実装を一本化する。
- **Enter 移動先**: activeCell 再ベースにより自然に解決（移動は再ベース後の index 基準）。編集中（Editing/Composing）のリモート構造変更は ime-editing-session の既存消失検知＋K4（DD-021-2）と整合させ、**編集セルの描画位置**は editor-placement の追従を確認。
- **Undo 生存整合（親①確定）**: 補償 SetCells の**実行前検査**（DD-020-3 の Codex P1b で導入済みの `validateOperation` 事前検査）が `target-row-deleted`/`unknown-row` を検出したらスタックから除去＋`undo-blocked`/`redo-blocked` 通知（既存語彙を流用・新語彙は原則なし）。挿入行は既存エントリへ無影響（beforeRevision 検査は revision ベースで行位置に依存しない）ことをテストで固定。
- **P2-1 是正（📐 で確定）**: `nextSlot` の全走査を **maxSlot の維持**（`SheetDocument` へキャッシュ追加 or rowMeta 更新箇所での単調更新）で O(1) 化。snapshot 復元・clone・un-tombstone 経路での maxSlot 整合が要点（decreasing しない=単調性維持で決定論を保つ）。`rowOrder.splice` O(N) は残る（1,000 連発でも要素移動のみ=実測で目標内なら許容・目標未達なら段階2としてギャップバッファ等を検討）。hash・snapshot round-trip・replay の決定論非退行を必須ガードとする。
- **計測計画（親⑤確定）**: ベンチ=50,000 行文書へ単一行 Insert×1,000 連発（committed 適用経路）で合計時間と per-op 中央値を計測し、行数を変えて線形性を確認。bulk 10,000 行 ≦500ms・replay 非退行（DD-014 計測系を流用）。回帰ガードは緩め（ヘッドルーム 4 倍以上）で常設・厳密値は親 Phase 4 実測記録（DD-020 paste-perf の運用に倣う）。

## 決定事項（親 要確認の確定〔2026-07-17〕の継承）

- **親①**: 行操作自体の Undo/Redo は対象外（計画書 §15.3 維持）。本子DDは Undo スタックの生存整合のみ（削除行エントリ=実行時拒否・除去＋通知）。対象化は Stage 3 バックログ候補。
- **親④**: 選択は生存行へ縮退・activeCell 削除時は最近傍生存行（下優先→上）へ移動。
- **親⑤**: 50,000 行×単一行 Insert×1,000 連発 合計 ≦2s かつ線形（1回 p95 ≦5ms）・bulk 10,000 行 ≦500ms 非退行・replay 非退行。
