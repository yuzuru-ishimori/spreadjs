## 目的

共同編集中でも整合が崩れない行の Insert/Delete を利用者機能として提供する（複数人が同時に行を挿入・削除しても全クライアントが同一状態へ収束する）。あわせて Stage 1 既知制約 K3（行挿入後の選択・Enter 移動先の再ベース）・K4（実IME 変換中の対象行削除）・P2-1（単一行 InsertRows 連発の Θ(N²)）を回収する（stage2-backlog §3・DD-018 確定）。

## 検討内容

### D1 スコープの重心（基盤は実装済み・新規は「利用者面×UI状態整合×性能」）
Operation・tombstone・収束・rollback・描画追従は Stage 1 資産として実装済み（上記実調査）。本DDの新規実装は (a) 公開 Command/API・UI 発火・standalone 対応・利用側通知契約、(b) 行構造変更×ローカル UI 状態（選択・activeCell・Enter 移動先・IME 編集）の再ベース=K3/K4、(c) P2-1 性能是正＋収束・統合検証の拡張、の3領域。これがそのまま子DD分割になる。

### D2 公開 API 形状（要確認⑥）
計画書 §5.4 は `sheet.execute({type:'insertRows', ...})` の Command bus を例示するが、現行 Facade（`GridInstance`）は個別メソッド流儀（`setData`/`focus`）。既定案は個別メソッド `insertRows`/`deleteRows` の追加（内部型非露出=R7 維持・execute bus 化は将来課題として境界化）。

### D3 行構造変更の利用側通知（standalone 保存契約・要確認⑥）
cell-commit はセル値専用（DD-024 契約）のまま維持し、行構造変更は新イベント種別（案: `row-structure-change`。挿入=rowId＋アンカー・削除=rowId。両モード共通）で通知する。単独グリッドモードはこれが保存材料になる。`setData` 再注入との整合（再注入は行構造ごと差し替え）も定義する。

### D4 同時 Insert の順序規則（要確認③）
計画書 §10.3 既定=**サーバー受付順・新 RowId で両方保持**。意図順の保証はしない（同一アンカーへの同時挿入は到着順で直列化・全クライアント同順に収束すればよい）。rejected にしない。

### D5 tombstone の永続化/GC（要確認②）
Stage 2 は GC しない（無期限保持）。snapshot format v1 が tombstone 行を保全済み・スロット再利用は「スナップショット再構築時のみ検討」（計画書 §6.3）を維持。Phase 4 の実測で問題が出た場合のみ ADR 起票。

### D6 Undo との整合（要確認①・計画書 §15.3）
行操作自体の Undo/Redo は **MVP後リスト（§15.3）どおり対象外**。ただし放置すると Undo スタックが削除行への補償 SetCells を送って不可解な reject になるため、DD-021-3 で整合のみ実装（削除行に触れる Undo/Redo エントリは実行時に生存検査で拒否・スタックから除去＋既存 `undo-blocked` 系通知。挿入行は影響なし）。

### D7 IME 編集中の対象行削除（K4・要確認④と関連）
計画書 §10.4 の精神（リモート更新は Document State へ適用・編集中ドラフトは変更しない・利用者が選ぶまで保持）に従い、既定案=編集は継続・commit 時に `target-row-deleted` 既存経路で reject 通知・ドラフトは textarea に保持（消さない）。実IME での確認は親 Phase 4 Manual Gate。

## 要確認（**全項目 確定済み 2026-07-17**・フル委譲モード=オーケストレータがユーザーOKを得て既定案どおり確定。確定値は各子DDの決定事項へ転記済み）

1. **要確認①: 行操作 Undo の対象化時期** — 確定（2026-07-17・既定案どおり）: **本DDでは対象外**（計画書 §15.3 MVP後リスト維持）。DD-021-3 は Undo スタックとの整合のみ実装（D6=削除行に触れるエントリの実行時拒否・除去＋通知）。対象化は Stage 3 バックログ候補として DD-032 棚卸しへ。
2. **要確認②: tombstone の永続化/GC** — 確定（2026-07-17・既定案どおり）: **Stage 2 は GC しない**（無期限保持・snapshot v1 は保全済み・スロット再利用はスナップショット再構築時のみ検討=計画書 §6.3 維持）。Phase 4 実測（削除多発文書の snapshot サイズ・復旧時間）で問題が出た場合のみ ADR 起票。
3. **要確認③: 同時 Insert の順序規則** — 確定（2026-07-17・既定案どおり）: **サーバー受付順・新 RowId で両方保持**（計画書 §10.3 既定のまま）。意図順（クライアント発行時刻等）の保証はせず、同一アンカー同時挿入の並びが到着順で決まることを仕様として文書化。rejected にしない。
4. **要確認④: 削除行を参照する数式/選択の扱い** — 確定（2026-07-17・既定案どおり）: 数式は未搭載（DD-022 未着手）＝**参照維持は対象外**とし既知制約に記録（数式導入時に tombstone 参照方針を定義）。選択は**生存行へ縮退**・activeCell の行が削除されたら**最近傍生存行（下優先→無ければ上）へ移動**。IME 編集中はドラフト保持＋commit 時 reject（D7）。
5. **要確認⑤: P2-1 性能目標** — 確定（2026-07-17・既定案どおり）: **50,000 行文書で単一行 Insert×1,000 連発のローカル適用合計 ≦2s かつ回数に対して線形**（Θ(N²) 解消の実証・1回 p95 ≦5ms）。bulk insert 10,000 行 ≦500ms（DD-014 実証水準の非退行）。replay（snapshot tail）経路の非退行も確認。
6. **要確認⑥: 単独グリッドモードの行操作 API と通知契約** — 確定（2026-07-17・既定案どおり）: `GridInstance` へ **`insertRows({ afterRowId, count })` / `deleteRows(rowIds)` 個別メソッド追加**（execute Command bus 化はしない=現行 Facade 流儀・R7 維持）。行構造変更は**新イベント `row-structure-change`**（両モード共通・standalone の保存材料。cell-commit はセル値専用を維持）。API 型 snapshot（DD-028）差分は CHANGELOG 記録。
7. **要確認⑦: UI 発火手段** — 確定（2026-07-17・既定案どおり）: **公開 API を正**とし、グリッド内蔵ショートカットは Excel 準拠 **Ctrl+Shift+'+'（アクティブ行の上へ挿入）/ Ctrl+'-'（選択範囲の行を削除）** を Navigation 位相のみで提供（Editing/Composing 中は無効=IME 不変条件維持）。コンテキストメニューは Stage 2 対象外（DD-027 以降の UI 拡張で再検討）。
