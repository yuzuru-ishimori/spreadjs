## 目的

Phase 0 の各PoC結果（DD-002〜006）・ADR・性能SLO・既知の制約・リスク再評価を**判定材料一式として集約・整理し、ユーザーに提示する**。Go／条件付きGo／No-Go の**判定はユーザーが行い**（自動継続ルールの停止条件「Go・No-Go判断」に該当）、本DDはその判定を記録する。Go（または条件付きGo）の場合、Phase 1 正式バックログ（ユーザー操作単位の縦切りDD一覧）を確定する。

**Goの定義**: 本DDにおける Go とは、**本番利用可能という意味ではなく、Phase 1 の製品化開発へ投資を継続してよいという技術成立性判断**である。PoC合格と製品完成を混同しない。

## 背景・課題

- 正典は計画書 `doc/plan/nanairo_realtime_spreadsheet_development_plan_v1.md` の **§18.6（Go/No-Go基準）・§19（Phase 1バックログ確定方針）・§21（性能SLO）・§23 Phase 0（完了条件）・§22（リスク表）**。作業管理は `doc/plan/phase0-dd-roadmap.md` の DD-007 行。
- **製品憲章 `doc/product/nanairo_sheet_product_charter_v1.md`（Accepted・DD-008）を Phase 1 バックログ確定の上位判断基準に加える**（外部レビュー第2回・助言3）。Go 後の Phase 1 は単なる機能開発ではなく、**製品成熟 Stage 1「社内SDK Alpha」（§15）＝Goal 1「社内の別プロジェクトから利用可能」（§30）を目指す段階**である。技術方式は承認済みADR・計画書が優先、製品目的・提供形態・非目標は製品憲章が優先（憲章 §1 の文書優先順位）。

- **着手条件: DD-002〜DD-006 全完了**（DD-006 のデータ表現・数式の計測結果も揃えてから判定する）。特に DD-002 の実機IME受入試験（Phase 6）と DD-004 のユーザー実機確認 run の結果は判定材料の必須項目。

## 検討内容

- **判定の構造**: エージェントは「材料の集約・§18.6基準との対照・所見の整理」までを行い、**Go/条件付きGo/No-Go の判定そのものはユーザーが行う**。エージェントが勝手に判定・続行しない（ロードマップ「自動継続のルール」の停止条件）。所見には各基準の達成/未達を機械的に対照した結果を付すが、最終欄（判定・条件・理由）はユーザー回答で埋める。
- **成果物は添付ファイルへ分離**（guides.md §6。50行超のため）: 判定材料一式 `DD-007/go-nogo-package.md`、Phase 1バックログ `DD-007/phase1-backlog.md`、Phase 1 SDK Alpha完了条件案 `DD-007/phase1-sdk-alpha-conditions.md`。本DDには判定記録と要約のみ残す。`go-nogo-package.md`（§1〜§8＋証拠レベル）と `phase1-sdk-alpha-conditions.md` は **DD-005待ち時間に空テンプレート/案として先行作成済み**（値・判定は未記入。Phase 1/2/3 で埋める）。
- **判定材料の構成**（受け入れ基準1〜6に対応）:
  1. PoC-A〜D 合格条件（§18.1〜18.4）× 実測値の対照表（出典: 各DDの計測レポート・試験証跡へのリンク付き）
  2. DD-005 統合シナリオ10項目の成立状況（項目別・証跡リンク付き）
  3. ADR一覧（§4 ADR-001〜024）の状態と、Phase 0 終了時が決定期限の ADR-005/008/011/022 の承認可否
  4. 性能SLO（§21 全14指標）× Phase 0 実測値の対照（未計測指標は「Phase 1以降で計測」と明記して暫定SLOのまま据え置くか確定するかを整理）
  5. 既知の制約一覧（各PoCが記録した簡略化・未検証事項・Phase 1引き継ぎ事項の集約）
  6. リスク表（§22 R-01〜R-20）の再評価（Phase 0 実証で低減したもの／残存するもの／新たに発見したもの）
- **判定項目の3分類**: §18.6基準との対照（go-nogo-package.md §7）では、全項目を同一のPass/Failにせず次の3種へ分類して提示する: **Hard Gate**（未達ならNo-Go。例: IME先頭文字欠落を安定回避できない・Operation非収束・ブラウザーメモリ上限超過）／**Conditional Gate**（制約付きGoが可能。例: ブラウザー限定・データ密度/列数制約）／**Observation**（Phase 1で継続測定。例: 性能SLO 14指標のうちPhase 0未計測の項目）。
- **証拠レベル（A〜E）を全判定材料の各根拠に付す**（外部レビュー第2回・助言1）: A=自動試験で再現可能（raw・seedあり）／B=headed実測（JSON・📸・環境情報）／C=ユーザー手動確認（詳細記録・トレース）／D=申告のみ・証跡なし／E=未実施。各基準の「合格」がどの強さの証拠に支えられているかを明示する。**Hard Gate を D または E だけで合格にする場合は、合格とせず「残存リスク」または「条件付きGo」として扱う**。特に DD-002 の4環境IME受入は当初「申告(D)」だったが、DD-005 の実IMEトレース回収で証拠レベルが上がるため、**DD-005実トレースでレベルが上がったかを §1/§2 に明示**する。
- **各PoCの結論は「確認できたこと／確認していないこと」を分離**する（助言2）: go-nogo-package.md §5（既知の制約）だけでなく §1 の各PoC結論欄でも分離し、**PoC成功を「製品完成」「全ネットワーク障害に対応済み」と表現しない**。例: DD-003 は server主導Operationでの収束・server→client欠落のcatch-up・冪等重複処理は確認済みだが、あらゆるclient→server欠落からの自動復旧・全障害パターン・本番認証認可を含むプロトコル安全性は未確認（DD-005 Non-Goals）。
- **条件付きGoの扱い**: §18.6 の条件付きGo（ブラウザー限定・データ密度/列数制約）に該当する場合、その条件を判定記録と Phase 1 バックログ（対象範囲・性能SLO）へ反映する。事前に許容範囲を決めず、材料を見てユーザーが判断する（要確認1回答: 承認）。条件付きGoの記録には**①条件 ②対象範囲 ③解除条件 ④期限 ⑤確認方法 ⑥未解除時の扱い ⑦Phase 1のどのDDをブロックするか**の7項目を必須とする（単なる問題の先送り化を防ぐ。DA調査②に対応）。
- **No-Goの扱い**: §18.6 の No-Go条件（IME先頭文字欠落を安定回避できない／rollback-replayの恒常遅延／メモリ上限超過）に該当した場合、Phase 3（バックログ）はスキップし、再設計の検討対象を整理して本DDを閉じる。再設計DD自体は別途起票する。
- **Phase 1バックログの原則**: ロードマップ「Phase 1以降: ユーザー操作単位の縦切りDD」に従う。**最初のDDは「日本語でセルを連続入力し、確定値が共同編集で保存される」で確定済み**（2026-07-12 ユーザー確定）。2件目以降の一覧・順序は §19 Phase 1 スコープと Phase 0 の既知の制約から起こし、ユーザーレビューで確定する（→要確認2）。
- **技術Go と Phase 1開始前提条件を分離して記録**（助言5・`DD-007/phase1-sdk-alpha-conditions.md` §4）: **技術Go**＝「製品化開発へ投資を継続してよいか」の成立性判断のみ。**Phase 1開始前提条件**＝採用CellStore／対応ブラウザー（Tier 1確定範囲）／残存IMEリスク／共同編集の既知制約／SDK Alphaの公開境界／Accepted化するADR／Phase 1で最初に解消する技術負債——を別リストとして記録する。技術Goでも前提を未整理のままPhase 1へ流さない。
- **最初の縦切りDDにSDK成立条件を含める**（助言4・`phase1-sdk-alpha-conditions.md` §2/§3）: 機能に加え、①`apps/` の内部ソースを直接importしない ②利用側は公開package exportだけを使う ③別consumerアプリからインストール・起動できる（`tests/consumer-vanilla`・`tests/consumer-react` または `apps/demo-sdk-consumer` fixture） ④ビルド済み成果物から利用できる ⑤公開Command/Event/Optionsで操作 ⑥内部CellStore/Canvasを露出しない ⑦Quick Start文書化 ⑧Reactなしで利用可 ⑨Reactラッパーは薄いAdapter——を完了条件とする（製品憲章 §15 Stage 1・§26.1）。方針は本DDで確定し、正式な完了条件化はPhase 3。
- **ADRはPoC成功でも機械的にAcceptedにしない**（助言6・go-nogo-package.md §3）: ADR-005/008/011/022 について、実証された前提／未実証の前提／既知の制約／再検討条件／製品憲章との整合を整理したうえで **Accepted／Proposed継続／条件付きAccepted** を判断する（状態欄の更新は要確認4のとおりPhase 2に含める。本文の大幅改稿はしない）。
- **必須項目のテンプレート反映**: ロードマップ「Phase 1以降の個別DDに必須とする項目」（19項目: 実現したい操作／対象範囲／対象外／状態遷移／Operation境界／競合／IME影響／Undo影響／性能影響／受け入れ基準／自動・手動検証／障害シナリオ／DA・Codexレビュー／証跡／既知の制約／ADR反映要否）を Phase 1 用差分テンプレートへ反映する**方針**を phase1-backlog.md に含める。テンプレート実ファイルの新設・改修は本DDの範囲か別DDかをユーザーが決める（→要確認3）。
- **スコープ外**: コード変更・計測の追加実施（不足計測が判明した場合は該当PoC DDの追補または別DDで実施し、本DDは集約のみ）／ADRファイル本文の大幅改稿（状態欄の更新は含む→要確認4）／Phase 1 個別DDの起票そのもの。
