## 目的

利用側供給の値→書式マッピング（背景色・バッジ）による view-local 書式描画を提供する（ReadyCrew=ステータス値のバッジ着色・stage2-backlog §3.5/§3.6）。あわせて書式共有化（Operation 化・snapshot 拡張）の設計整合文書を作成し（実装は発火条件付き・親④）、C級のダブルクリック auto-fit（backlog §3.5）を回収する。

## 背景・課題

- 親 DD-027 の決定事項（転記・2026-07-21 ユーザー確定）:
  - **④**: セル書式（背景色・バッジ）は**利用側供給の値→書式マッピングによる view-local 描画**。Operation 化・snapshot 拡張（全ユーザー共有）は**設計整合（モデルの置き場・拡張方針の文書化）まで**を本DDで行い、実装は**発火条件付き**（共同編集採用案件の確定で実装子DDを採番。Stage 2 中に不成立なら DD-032 が Stage 3 バックログへ登録=roadmap §7）。
  - **⑤**: 公開は宣言的 mount オプションのみ・Internal registry は非公開。カスタム描画 callback 等の Plugin 面は P-07 材料（親 Phase 4 メモ）へ。
- DD-027-1/2 で成立済み・進行中の基盤（実調査・二重設計しない）: `column-types.ts`（registry・`GridColumnType` union・fail-fast）・mount-controller の registry 生成/配線・base-layer の列単位フック先例（`isWrapColumn`・27-2 で `isLinkColumn` 追加見込み=DD-027-2 📐）・playground URL パラメータ/E2E 先例。
- 描画経路の実調査（2026-07-21）:
  - `packages/render/src/base-layer.ts` `drawPane`: pane 一括の背景 fillRect → `drawPaneGrid`（罫線）→ `drawPaneValues`（**`store.queryRange` visitor で可視非空セルのみ**を描画。数値=右寄せ／wrap=`drawWrapCell`／左寄せ文字列=オーバーフロー延長＋左外流入 Pass 2）。セル単位の背景・バッジ描画は存在しない。罫線の**後**に values を描くため、セル背景をここで塗るなら罫線を保存する工夫が要る。
  - `packages/render/src/text-cache.ts`: measureWidth/fitText/wrapLines の FIFO 上限付きキャッシュ（base-layer と自動行高で共有）。バッジのチップ幅測定にそのまま使える。
  - `packages/grid/src/document-view.ts`: read-through `store`（ClientSession 文書を都度読む・セル状態非保持）。`setColumnWidth`（override 管理・既定値で解除・DD-012-4）が auto-fit の書き込み先。
  - `packages/grid/src/resize-interaction.ts`: `resizeHitTest`（列境界の掴み代判定・純関数）・`clampColumnWidth`（20〜2000px）。dblclick での auto-fit 先取りにそのまま使える。mount-controller の dblclick ハンドラは現状 `hit.area==='cell'` のみ処理（ヘッダー境界 dblclick は未使用=衝突なし）。
  - 計測ハーネス: `apps/playground/src/integration/perf-capture.ts`（`?perf=1`・frame p50/p95・stoppedRedrawMs）＋ `scripts/cg-perf/run-cg6.mjs`（headed Chrome・judge=`judge-perf-report.mjs`・予算=`perf-budget.json`）。DD-012-4/5 の「headed perf 回帰なし」と同一経路で before/after 比較できる。
- DA 論点（親 Phase 0 より・本DDで扱う分）: (a) **view-local の「共有されているように見えて各自設定」リスク** — 書式は値ベース（同じ値なら同じ見た目）＋設定は利用側配布のため、設定が一致していれば実質同一表示。ただし**設定不一致クライアントは異なる装飾を見る**（文書状態は不変・hash 乖離なし）。利用側ガイドと共有化設計文書へ明記。 (b) **フレーム予算** — 書式 lookup を描画ホットパス（可視非空セル×毎フレーム）に入れるため、O(1) lookup（プリコンパイル済み Map）＋非空セル限定で設計し、headed 計測で判定。 (c) **text-cache との干渉** — バッジは fitText/measureWidth を共有キャッシュで使う（キー体系は既存のまま=干渉なし。エントリ増は FIFO 上限内）。 (d) **オーバーフロー/wrap との重なり順** — 下記📐で規定。

## 📐 実装前詳細化（親 Phase 0 判定=要。2026-07-21 実施済み・👀 ユーザーレビューは実装開始前の確認ゲートで）

**結論: 書式は「値の完全一致ルール」を mount 時にプリコンパイルした Map（列→値→style）で O(1) 解決し、base-layer の既存 queryRange visitor（可視非空セルのみ）に1分岐だけ足す。編集・IME・commit 経路には一切触れない＝T1 非該当。**

- **書式 API 形状（公開・宣言的）**:
  ```ts
  interface GridColumnFormatRule {
    readonly match: string | readonly string[];   // セル表示文字列の完全一致（v1。範囲・正規表現・callback は拡張点メモ→P-07）
    readonly style: {
      readonly cellBackground?: string;           // セル背景色（CSS color・罫線は保存=inset 塗り）
      readonly textColor?: string;                // 文字色
      readonly badge?: boolean;                   // true=値を丸角チップ（バッジ）で描画
      readonly badgeColor?: string;               // チップ背景色（badge:true 時。既定は cellBackground 系）
    };
  }
  // GridCommonMountOptions へ追加（両モード共通・mount 時固定＝columnTypes と同運用・親⑥）
  readonly columnFormats?: Readonly<Record<string, readonly GridColumnFormatRule[]>>;  // キー=ColumnId 文字列・ルールは先勝ち
  ```
  fail-fast（`column-types-invalid` と同系の config error）: 未知列・空ルール配列・match 重複。色文字列の妥当性は検査しない（canvas fillStyle は不正値を無視＝安全）。
- **描画接続点**: 新規 `packages/grid/src/format-rules.ts`（公開型＋`compileFormatRules`→列→値→resolved style の Map・純関数=TDD）。base-layer へ `getCellStyle?: (colIndex, value) => ResolvedCellStyle | undefined` フックを追加（`isWrapColumn`/`isLinkColumn` と同型・DOM 非依存・grid 側が Map lookup を束縛）。
  - **背景色**: queryRange visitor 内で style があれば**セル矩形から罫線幅ぶん inset した fillRect**を文字より先に塗る（既存の grid→values 描画順を変えず罫線を保存・1 pass 維持）。
  - **バッジ**: 値を丸角チップ（badgeColor 塗り＋textColor 文字・左右 padding・fitText で単行クリップ）で描画。**バッジ指定値のセルはオーバーフロー対象外**（チップ描画が崩れるため・リンク列と同じ裁定=DD-027-2）。
  - **重なり順の規定**: 書式は**非空セルのみ**に付く（値ベース）ため、空セル=書式なし=オーバーフロー流入先のまま／非空セル=元々 blocked＝**左隣のオーバーフロー文字と書式背景の衝突は構造的に起きない**。wrap 列×背景色=可（rect 塗り→wrap 文字）。wrap 列×バッジ=バッジが勝ち単行チップ（wrapLines を通さない）。
  - **数値セル**: 書式一致すれば textColor が numberText 既定色より優先（右寄せは維持）。
- **性能設計と計測方法**: lookup は Map（プリコンパイル・ルール線形走査を描画時にしない）＋可視非空セル限定＝追加コスト O(可視非空セル)。判定は既存ハーネスで before/after:
  1. `bash scripts/dev-start.sh`（50k 行シード）
  2. `node scripts/cg-perf/run-cg6.mjs http://localhost:5885 http://127.0.0.1:9499`（headed・`?perf=1`。書式あり計測は playground `?format=...` パラメータを追加して同 URL で実行）
  3. `judge-perf-report.mjs`（`perf-budget.json`）で frame p95 ≤16.7ms・stoppedRedrawMs ≤12ms（境界値）を判定・レポートを `doc/DD/DD-027-3/` へ保存
- **auto-fit 設計（C級・DD-012-4 資産へ接続）**: mount-controller の dblclick ハンドラ冒頭で `resizeHitTest` を先取り（現状ヘッダー境界 dblclick は未使用=既存 `doubleClickCell` と衝突しない）。列境界 dblclick →
  - 対象列の非空セルを `view.store.queryRange(0, rowCount, col, col+1)` で走査し、text-cache `measureWidth` の最大値＋padding（バッジ指定値はチップ幅）を求める。**列ヘッダーラベル幅も含める**（Excel 準拠）。
  - **上限ガード**: 走査 10,000 非空セルで打ち切り（それまでの最大値を採用＋診断 info・50k 行列の単発操作でも予算内に収める）。
  - `clampColumnWidth` → `view.setColumnWidth` → **layout イベント発火**（DD-012-4 D2 の確定通知・利用側保存契約を維持）。**wrap 列は対象外**（折り返し前提の列に内容幅 fit は無意味＝診断 info・無変更）。
- **共有化設計整合文書（確定④の「設計まで」・実装しない）**: 成果物 `doc/plan/cell-format-sharing-design.md`（新設・DOC-MAP 登録）。内容: 書式の文書プロパティ化の設計（FormatOperation の形・snapshot format 拡張と version 方針・正準 hash への含め方・OCC 粒度・view-local ルールとの合成順序・**列幅/行高/wrap の全ユーザー共有〔backlog §3.5〕との統合方針**・発火条件と実装子DD採番の手順）。
- **モジュール構成（新規1・変更4＋文書1）**: 新規 `packages/grid/src/format-rules.ts`／変更 `packages/grid/src/index.ts`（columnFormats＋公開型）・`packages/grid/src/mount-controller.ts`（rule index 生成 fail-fast・base-layer 束縛・auto-fit dblclick 先取り・layout emit）・`packages/render/src/base-layer.ts`（getCellStyle フック・inset 背景・バッジチップ・数値色優先）・`apps/playground/src/integration/main.ts`（`?format=` 配線）／新規文書 `doc/plan/cell-format-sharing-design.md`。document-view は変更なし見込み（値は read-through store 経由で届く）。
- **テスト観点**: compileFormatRules（先勝ち・完全一致・fail-fast）＝unit／描画（背景 inset・バッジ・重なり順・数値右寄せ維持）＝E2E＋エビデンス／予算=headed 計測 before/after／auto-fit（最大幅・ヘッダー含む・上限打ち切り・clamp・layout イベント・wrap 除外）＝unit＋E2E。

## 検討内容

- **一致条件の v1 範囲**: 完全一致（`match: string | string[]`）のみ。値レンジ（数値比較）・正規表現・利用側 callback は「描画 Plugin のフレーム予算契約」（憲章 §13.2）が要るため v1 に含めず、拡張点メモとして親 P-07 材料へ（新規論点として確認ゲートに提示）。
- **列全体の背景色（値によらない静的な列色）**: v1 に含めない（書式は値ベースのみ。ReadyCrew 要件に無く、空セルへの書式はオーバーフロー流入先の裁定を複雑化する）。要求が出たら共有化設計文書の拡張方針に従い追加。
- **バッジの形**: 値テキストをそのまま丸角チップで描く（アイコン・任意テキストの分離はしない=値1本の原則・親③と整合）。

## 決定事項

- 親④⑤（背景に転記）＋📐 の設計（宣言的 `columnFormats`・完全一致 v1・プリコンパイル Map・非空セル限定・inset 背景・バッジ=オーバーフロー対象外・auto-fit 上限ガード付き・共有化は設計文書のみ）を採用。
- 公開面: `columnFormats` mount オプション＋公開型（Experimental 0.x）。新規イベント・reject 語彙なし（auto-fit は既存 layout イベント）。API 型 snapshot 差分は CHANGELOG へ記録。

## 既知制約（DD-027-3・v1）

- **wrap 列 × バッジ書式の行高不整合（Fable P3-1）**: wrap（折り返し）列のセル値にバッジ書式が一致した場合、描画は単行チップ（バッジは自セル内単行クリップ＝📐 規定）だが、自動行高計算は wrapLines ベースで折り返し行数ぶんの高さを維持するため、「背の高い行にバッジ 1 行チップだけ」が表示されうる。v1 は実害小（誤情報なし・レイアウト崩れなし）ゆえ**未対応**。将来「バッジ一致値を自動行高計算から除外」する拡張点を共有化設計文書 `doc/plan/cell-format-sharing-design.md` の拡張方針側で扱う（実案件要求が発火条件）。
- **一致条件は表示文字列の完全一致のみ（v1）**: 値レンジ（数値比較）・正規表現・利用側 callback・列全体の静的背景色（値によらない列色）は v1 対象外。「描画 Plugin のフレーム予算契約」（憲章 §13.2）を要するため P-07 判断材料メモ（親 Phase 4）へ拡張点として送る。
- **書式は view-local（発火条件付きで共有化）**: 全ユーザー共有（Operation 化・snapshot 拡張）は設計整合文書のみ作成し実装は保留（親④・共同編集採用案件の確定が発火条件）。設定不一致クライアントは異なる装飾を見るが文書 hash は乖離しない。
- **standalone の `?format=` は playground 配線済み・描画経路は両モード共通（Fable P3-4）**: 単独グリッドモードの columnFormats は mount-controller の `buildColumnTypeRegistry`→`getCellStyle` 束縛が collab モードと同一経路（`attachBackendRendering` 共有）ゆえ担保される。standalone 専用の書式 E2E は追加していない（select/link と同じく共有経路＋integration モード E2E でカバー）。
