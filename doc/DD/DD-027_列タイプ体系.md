## 目的

列単位で「編集方式・クリック挙動・表示装飾」を変えられる**列タイプ体系**を SDK 機能として提供する（選択式入力列・ハイパーリンク列・セル書式〔背景色・バッジ〕）。あわせて列タイプを **Cell type plugin の実質プロトタイプ**として設計し、P-07（Plugin API v1 範囲・DD-030 起票前の独立判断ゲート）の判断材料を作る（roadmap §1/§2/§4 R1・S2-6）。

## 背景・課題

- 正典: roadmap §1 DD-027 行（**アンブレラ・子3本指定**: 27-1 選択式入力列・27-2 ハイパーリンク列・27-3 セル書式モデル。Risk Class A・Human Spec Gate 必須・ReadyCrew 要件・P-07 判断材料・S2-6）・stage2-backlog §3.5（セル単位書式モデル・列幅等の全ユーザー共有=発火条件付き・ダブルクリック auto-fit=本DD配下の C級タスク）／§3.6（3機能の出典=ReadyCrew 案件DB 商談進捗パイプライン画面・「新しいアーキテクチャ概念」判定）・計画書 §2.3（MVP データ型に「インライン定義のプルダウン」、表示に「基本的な背景色、文字色、配置、罫線」を含む）・製品憲章 §13.1（Plugin 候補: Cell type plugin／Value parser・formatter／Validator／Canvas renderer・overlay／Cell editor）・§13.2（**Plugin API v1 は複数実案件の共通要求が確認されるまで確定しない**・描画 Plugin はフレーム予算を破らない契約）。
- 既存実装の実調査（2026-07-21・本DD起票時）:
  - **値モデル（core・変更しないことが論点①）**: `packages/core/src/operations.ts` の `CellScalar` は blank/string/number/date の4種（正準 hash・replay 決定性・snapshot v1 に関与）。`cell-input.ts` の `parseCellInput` が確定ドラフト文字列→CellScalar の唯一の正準変換（commit 経路が委譲・IME 非関与の純関数）。
  - **描画（背景色・バッジは未実装）**: `packages/render/src/base-layer.ts` はセル背景=全セル一律 `cellBackground`・文字色は `isNumericCell` による numberText/cellText の2択のみ。セル/列単位の背景色・バッジ・装飾の描画経路は存在しない。`packages/grid/src/document-view.ts` は純読み取りアダプタ（セル状態非保持・表示文字列は `cellScalarToDisplay`）。
  - **公開面（列単位設定の先例あり・列タイプは無い）**: `packages/grid/src/index.ts` の `GridCommonMountOptions` に `columnWidths`/`rowHeights`/`wrapColumns`（列単位・mount 時固定・view-local=DD-012-4/5 の先例）。イベントは layout/cell-commit/row-structure-change 等。列タイプ・候補リスト・リンク・書式の API/イベントは一切ない。
  - **列メタの持ち方**: 列は固定（`columnOrder`=ColumnId 文字列列。共同編集=server /config・単独=mount オプション必須）。列単位の型・候補・書式を持つメタ構造は未実装。
  - **クリック**: `mount-controller.ts` の pointerdown→`editor.pointerdownCell`（選択・編集確定）＋ドラッグ選択、dblclick→編集開始。クリックを「選択でなく別ナビゲーション」と解釈する経路は無い。
  - **編集**: 常駐 textarea＋editor-state-machine（`packages/ime`）による自由入力1種類のみ。ドロップダウン等の代替エディタ機構は無い。
  - **切り分け**: 本DDの対象（列タイプメタ・選択式エディタ・リンククリック解釈・書式描画）は**全て新規**。再利用する既存資産=原子 SetCells＋OCC・確定単位 chokepoint `submitSetCells`（DD-020）／公開語彙運用=CHANGELOG・`doc/DD/DD-017/error-codes.md`・API 型 snapshot（DD-021/028）／列単位 mount オプションと layout イベントの先例（DD-012-4/5）／text-cache・overlay 描画基盤。
- 検証の教訓（DD-021 で確立・本DDの検証方針として全子DDへ適用）: (1) 統合検証は playground E2E を**全スイート**で回す（挙動置換の移行漏れ検出）。(2) E2E で view 状態（rowIdAt/rowCount 等）を読む箇所は `expect.poll` でゲート（view 軸は描画 flush 時再構築）。(3) 共有文書を大量変更する spec は `zz-` prefix でスイート最後に隔離。

### D2〜D7（要確認①〜⑥の検討・既定案の根拠）

- **D2（①モデル置き場）**: CellScalar 拡張は hash 決定性・snapshot v1・OCC・protocol の全経路へ波及する（Risk A の protocol 変更）。一方 3機能とも「値は string のまま・列単位で編集方式/クリック/描画を変える」で成立する。憲章 §13.1 の Plugin 候補（Cell editor・Validator・renderer）とも「grid 層の登録制メタ＋フック」が一致。
- **D3（②候補供給・検証）**: 計画書 §2.3 は「インライン定義のプルダウン」= 静的候補。動的供給（callback）は共通要求が未確認（§13.2）なので拡張点の設計メモに留める。検証は commit 前（chokepoint `submitSetCells` の手前）の列 validator。サーバーは列タイプを知らない（grid 層メタ）ため、paste/setData/他クライアント由来の非候補値は**拒否せず保持・表示**（サイレント破壊・収束乖離を作らない）。
- **D4（③リンク値・安全性）**: 値は string 1本（URL または任意テキスト）。SDK の既定動作を「navigate せず `link-open` イベント通知のみ」にすれば、SPA 内遷移（ReadyCrew の詳細画面）に正しく委ねられ、`javascript:`/`data:` スキームの実行も SDK 内では発生しない。opt-in の既定 open（window.open）を提供する場合のみ http/https 限定。
- **D5（④書式の共有性）**: roadmap §7 の確定どおり「設計整合まで実施・Operation 化/snapshot 拡張は発火条件付き（共同編集採用案件の確定）」。Stage 2 consumer 2件は共同編集不採用（roadmap §6）のため、利用側供給の表示ルール（値→書式マッピング）で実質的に全員同一表示にできる。DD-012-4 D1（view-local＋利用側保存）の先例と整合。
- **D6（⑤Plugin 境界）**: 憲章 §13.2 により v1 確定は禁止。内部 registry を「v1 候補インターフェース」として設計し、P-07 判断ゲートへ材料（memo）として提出する。consumer 向け register API は公開しない。
- **D7（⑥単独モード・通知契約）**: 列タイプ・書式ルールは両モード共通の mount オプション（`wrapColumns` と同じ mount 時固定運用）。cell-commit/setData の値契約（表示文字列 round-trip）は不変。共同編集での設定一致は利用側責務（/config への統合は要求が出たら別途）。

## 決定事項

- **Human Spec Gate 確定（2026-07-21・ユーザー承認「進めてください」＝要確認①〜⑥すべて既定案どおり採用）**:
  1. 列タイプのモデル置き場 = **grid 層の列タイプメタ**（mount オプション `columnTypes`＋内部 ColumnTypeRegistry）。core の CellScalar・protocol・snapshot は変更しない（値は string のまま）。
  2. 選択式入力 = **列ごとの静的候補リスト**・検証は **commit 前（`submitSetCells` 手前）の列 validator**・自由入力は既定不可（`allowFreeText` で許可可）・**paste/setData/リモート由来の非候補値は保持・表示**（拒否しない）。動的供給は拡張点メモのみ。
  3. ハイパーリンク = 値は **string 1本**・既定動作は **navigate せず `link-open` イベント通知のみ**・opt-in の既定 open を提供する場合は http/https 限定・クリック時は activeCell 移動従来どおり＋`link-open` 発火・編集手段（F2/ダブルクリック等）維持。
  4. セル書式（背景色・バッジ）= **利用側供給の値→書式マッピングによる view-local 描画**。Operation 化・snapshot 拡張は 27-3 で**設計整合まで**（実装は発火条件付き=共同編集採用案件の確定。不成立なら Stage 3 バックログへ）。
  5. Plugin 境界 = **公開は宣言的 mount オプションのみ**。ColumnTypeRegistry は Internal として設計・consumer 向け登録 API 非公開。親 Phase 4 で P-07 判断材料メモを作成し P-07 ゲートへ提出。
  6. 単独グリッドモード = `columnTypes`・書式ルールは**両モード共通の mount オプション（mount 時固定=wrapColumns と同運用）**。cell-commit/setData の値契約不変・新規公開イベントは `link-open` のみ・共同編集での設定一致は利用側責務。
- 確定値は各子DD（27-1: ①②⑤⑥／27-2: ③⑥／27-3: ④⑤）の決定事項へ転記する。

## 要確認（Human Spec Gate・各項目に既定案）

1. **要確認①: 列タイプのモデル置き場** — 既定案: **grid 層の列タイプメタ**（mount オプション `columnTypes`＋内部 ColumnTypeRegistry）とし、core の CellScalar・protocol・snapshot は**変更しない**（値は string のまま）。根拠=D2。対案: core 値モデル拡張（kind: 'link' 等）は hash/snapshot/OCC へ波及し Stage 2 の要件を超える。
2. **要確認②: 選択式入力の候補供給と検証タイミング** — 既定案: **列ごとの静的候補リスト**（mount オプション・計画書 §2.3「インライン定義のプルダウン」準拠）。検証は **commit 前（`submitSetCells` 手前）の列 validator**。自由入力は既定不可（列オプション `allowFreeText` で許可可能）。**paste/setData/リモート由来の非候補値は保持・表示**（拒否しない=収束・データ非破壊優先）。動的供給（callback/Promise）は対象外・拡張点メモのみ。
3. **要確認③: ハイパーリンクの値表現と安全性** — 既定案: 値は **string 1本**（表示テキスト＋URL の分離モデルは導入しない）。クリック時は **navigate せず公開イベント `link-open`（rowId/columnId/value）で利用側へ委ねるのが既定動作**。opt-in の既定 open を提供する場合は **http/https のみ**（javascript:/data: 等は診断 warn＋無視）。クリック競合の裁き: リンク列のクリックも **activeCell 移動は従来どおり**行い＋`link-open` を発火（選択を奪わない）。編集開始は F2/ダブルクリック/直接入力の既存手段を維持。
4. **要確認④: セル書式（背景色・バッジ）の共有性** — 既定案: **利用側供給の表示ルール（値/セル→背景色・バッジのマッピング）による view-local 描画**。文書プロパティ化（Operation 化・snapshot 拡張=全ユーザー共有）は**設計整合（モデルの置き場・拡張方針の文書化）まで**を 27-3 で行い、実装は**発火条件付き**（共同編集採用案件の確定で実装子DDを採番。Stage 2 中に不成立なら DD-032 が Stage 3 バックログへ登録=roadmap §7）。
5. **要確認⑤: Plugin API v1 として切り出す境界** — 既定案: **公開は mount オプション（宣言的設定）のみ**。editor/renderer/validator のフック集合（ColumnTypeRegistry）は **Internal として設計**し、consumer への Plugin 登録 API は公開しない。親 Phase 4 で **P-07 判断材料メモ**（registry の形＝v1 候補 I/F・設定だけで実現できた/できなかった要求・fork 圧力の記録）を作成し、P-07 判断ゲート（DD-030 起票前）へ提出する。根拠=憲章 §13.2。
6. **要確認⑥: 単独グリッドモードでの列タイプ設定 API と通知契約** — 既定案: `columnTypes`・書式ルールは **両モード共通の mount オプション（mount 時固定）**。cell-commit/setData の値契約は不変（選択式・リンクとも値は表示文字列で round-trip）。新規イベントは `link-open` のみ（両モード共通）。共同編集モードでの全クライアント設定一致は利用側責務（server /config への統合は Stage 2 対象外）。
