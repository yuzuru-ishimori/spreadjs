# DD自動フロー（起票=Fable → 〔確認〕→ 実装=Opus → Codexレビュー）

DDの一連の流れを、各フェーズを専門サブエージェントに委譲して自動実行するスキル。
**あなた（このセッションの主ループ）はオーケストレータ**であり、重い処理は自分でやらず各モデルに振り分ける。

> **前提となる仕組み**: Claude Code は主ループのモデルを実行中に切り替えられない。そこで
> 「起票=Fable」「実装=Opus」を**サブエージェントのモデル指定**で実現する。主ループのモデルは何でもよい
> （オーケストレーションの軽い処理のみ）。Codex は外部CLI（`scripts/codex-review.sh`）で呼ぶ。

## 手順

### Step 1: 起票（Fableサブエージェント）

`Agent` ツールを `subagent_type: "dd-drafter"` で1体起動する（モデルはエージェント定義側で Fable 固定）。
プロンプトにユーザーの「作りたい機能・変更の要望」を、背景・制約・希望も含めてそのまま渡す。

- 戻り値（DD番号・目的・スコープ・主要タスク・Codex要否・**要確認事項**）を受け取る。
- 起票が失敗（テンプレート未検出など）した場合はユーザーに報告して停止する。

### Step 2: 仕様確認ゲート（必須・ここで必ず止まる）

Step 1 の戻り値を整形してユーザーに提示し、**明示的なOKを得るまで実装に進まない**。

```
📋 DD-{番号} を起票しました（Fable）。この仕様で実装（Opus）に進んでよいですか？
- 目的 / スコープ: ...
- 主要タスク: ...
- Codexレビュー: {必須/推奨/不要} (effort={high/xhigh})
- 要確認事項: ...   ← ある場合はユーザーの判断を仰ぐ
修正があれば指示してください。「OK」「進めて」で実装に入ります。
```

- ユーザーが**修正を指示**したら: 軽微なら主ループで DD を Edit、方針レベルの変更なら
  `dd-drafter` を修正指示付きで再起動する。再提示して再度OKを得る。
- ユーザーの**OKが得られるまで Step 3 に進まない**。

### Step 3: 実装 + Codexレビュー（Opusサブエージェント）

`Agent` ツールを `subagent_type: "dd-implementer"` で1体起動する（モデルは Opus 固定）。
プロンプトに **DD番号** と、Step 2 でユーザーが合意/修正した内容を明示して渡す。

- 実装エージェントが実装・テスト・Codexレビュー・findings反映・DDログ更新まで行う。
- 「要判断」で止まって戻ってきた場合は、その論点をユーザーに提示して判断を仰ぎ、
  必要なら追加指示を付けて `dd-implementer` を再起動する。

### Step 4: 完了報告

実装エージェントの戻り値を要約してユーザーに提示する：

```
✅ DD-{番号} 実装完了
- 変更: ...（ファイル/タスク要約）
- テスト: {green/red 要点}
- Codexレビュー: findings {n}件 / 対応 {n}件 / 見送り {n}件（理由）
- 要判断・残課題: ...

次のステップ: 内容を確認し、問題なければコミット → `/dd archive {番号}` でクローズできます。
```

- **コミット・アーカイブはユーザー確認後**。このスキルは自動でコミットしない。

## 注意

- 主ループのモデルは問わない（安いモデルでオーケストレーションしてよい）。実際のコスト配分は
  サブエージェント側（起票=Fable / 実装=Opus）で決まる。
- Step 2 の確認ゲートは**スキップ禁止**。完全ノーストップにしたい場合はユーザーの明示指示があるときだけ。
- 各サブエージェントはユーザーに直接質問できない。確認が必要な点は必ずこのオーケストレータ経由で
  ユーザーに提示する。
